Architecture Learning Document

Memory Architecture in Agentic Systems

A working architect's notes on building production memory for AI agents — working, episodic, semantic, and procedural — across Spring AI agents, batch jobs, and the broader enterprise platform.

Audience
Platform, AI, and Data engineering teams
Scope
Conceptual foundations + implementation guidance
Status
Living document — patterns evolving
Framework versions assumed

Patterns and code in this document are anchored to: Spring Boot 3.4+ / 4.0, Spring AI 1.1.x (stable). Memory artifact IDs follow the post-1.0 RC1 naming convention (spring-ai-model-chat-memory-repository-*). PromptChatMemoryAdvisor is deprecated since 1.1.3 — MessageChatMemoryAdvisor is the recommended path. PGVector is assumed as the default vector store given existing PostgreSQL footprint. NIM models are reached via the LiteLLM proxy; embedding model is provider-agnostic at the Spring AI layer.

§ 01

Context and Foundations

An LLM is stateless. Every call is a fresh universe: the model has no recollection of the previous turn, no awareness of which user asked, no memory of what worked last time. State that feels continuous to the user is an illusion the application maintains by re-presenting context on every call. Memory architecture is the discipline of deciding what to re-present, when, and from where.

For deterministic services, this discipline has a familiar name — session management, caching, knowledge bases — and the patterns are well understood. Agents force the conversation into harder territory: the LLM both reads from and writes to memory, the “right” thing to remember is not knowable in advance, and an agent that runs autonomously over hours or days needs memory that genuinely persists across sessions, processes, and even infrastructure failures.

The patterns established to solve memory for agents are not narrowly applicable to AI workloads. They are modern data patterns — typed event streams for episodic recall, vector search for semantic retrieval, versioned configuration for procedural rules, governance hooks for retention and auditability — that any stateful service benefits from. The AI use case forces the architectural conversation; the architecture benefits every workload that follows.

Memory is not a feature of an agent. It is the substrate the agent runs on. Choose its shape deliberately, or live with whatever shape emerges by accident.

This document captures the architectural reasoning, the Spring AI primitives available for each memory type, the patterns for driving memory programmatically outside the ChatClient (essential for batch jobs and scheduled workflows), and the governance hooks SOX-regulated environments require. As with identity, the same reasoning applies to whichever agent comes next.

§ 02

The Cognitive Memory Model

The vocabulary worth standardizing on comes from CoALA (Cognitive Architectures for Language Agents, Sumers et al., 2023), which translated Tulving's 1972 distinction between episodic and semantic memory — and Squire's 1987 addition of procedural memory — into a clean model for LLM-driven systems. Major frameworks (LangChain, LangGraph, Letta, Mem0, IBM, MongoDB) all use a version of this taxonomy in their documentation. The team should too.

CoALA describes an agent with a short-term working memory plus three optional long-term memories: episodic (experiences), semantic (knowledge), and procedural (skills). Each has different storage characteristics, different read/write patterns, different governance needs, and different roles in the decision cycle. Treating them as one — the most common architectural mistake — produces an agent that retrieves the wrong thing at the wrong time and audits nothing reliably.

Why this taxonomy matters

The taxonomy is not academic. It maps directly to which storage backend is correct for what. Conversation turns belong in a chat-memory repository keyed by conversation ID, not a vector store. Domain knowledge belongs in a vector store with metadata filters, not in chat history. Business rules belong in version-controlled prompt files and a curated tool catalog, not embedded as facts in either. Confusing these collapses governance, retrieval quality, and cost.

The remainder of this document treats the four memory types individually before showing how they compose in a single agent — and crucially, how each can be driven programmatically without a ChatClient in the loop.

§ 03

The Four Memory Types

Every agent participates in four memory planes simultaneously. The mental model to hold: what does the model see on this call, what did it see on the last call, what does it know about the world, and how does it act.

PLANE 01
Working Memory
The current context window. Everything the LLM literally sees on this call. Built fresh each turn from the other three. Bounded by token budget; not persisted.
PLANE 02
Episodic Memory
Conversation turns and event logs — what happened. Keyed by conversation/session ID. Spring AI: ChatMemory + ChatMemoryRepository.
PLANE 03
Semantic Memory
Domain knowledge and consolidated facts — what is known. Embedded and retrieved by similarity. Spring AI: VectorStore + EmbeddingModel.
PLANE 04
Procedural Memory
Skills and rules — how to act. System prompts + tool catalog. Spring AI: @Tool methods, system prompt files, tool callbacks.
Memory Composition at Inference Time
1
Episodic
Load recent conversation turns for this session (e.g., last 20 messages from chat_memory table keyed by conversation ID).
2
Semantic
Embed the user's input. Run similarity search against the vector store with metadata filters for tenant/user/agent scope. Pull top-K relevant chunks.
3
Procedural
Load the agent's system prompt (versioned in Git). Resolve tool callbacks from the registry. Optionally filter tools by tenant policy.
4
Working
Compose into a single Prompt: system message (procedural) + semantic context block + episodic message history + current user message. Send to the model.
5
Write-back
After the response: append user + assistant turn to episodic store. Optionally trigger consolidation (extract facts → semantic store) and procedural learning (record tool success/failure).

Working memory is ephemeral and free; the other three are persistent and have storage, governance, and lifecycle implications. Most of this document is about those three.

§ 04

Working Memory

Working memory is the model's context window for a single inference call. It is composed fresh from the long-term stores, exists only for the duration of one round trip, and is bounded by the model's token budget. The architectural decisions about working memory are really decisions about the composition strategy: in what order, with what budgets, and via which truncation policy do the long-term stores assemble themselves into a prompt?

Composition order matters

The conventional order — and the one Spring AI's advisor chain enforces by default — is:

  1. System message (procedural) — agent role, rules, tool-use policy
  2. Retrieved context block (semantic) — relevant knowledge for this query, prefaced with a clear separator
  3. Conversation history (episodic) — last N message pairs
  4. Current user message — the turn's input

This ordering matters because of how attention works in practice: tokens near the start (system prompt) and near the end (current question) receive the most reliable attention. Recent guidance — “lost in the middle” research — suggests keeping the most critical context out of the middle of long prompts.

Token budgeting

Each plane should have a declared budget, not a soft preference:

PlaneTypical budgetTruncation policy
System prompt500 – 1,500 tokensNone — fixed content
Semantic retrieval2,000 – 4,000 tokensTop-K with similarity threshold; trim chunks
Episodic history2,000 – 6,000 tokensSliding window: keep last N messages
Current turn + headroom for responseRemainderReserve ≥ max_tokens for generation
The silent failure mode

The model never tells you when context was truncated. A long conversation with a tight window quietly drops the oldest turns from MessageWindowChatMemory's sliding window — the user references something from earlier and the agent acts confused. Either size the window deliberately for expected turn length, or use a summarization strategy (consolidate older turns into a single summary message) for genuinely long sessions.

Spring AI's MessageWindowChatMemory handles the sliding window for the episodic plane. For the semantic plane, the SearchRequest parameters (topK, similarityThreshold) shape the budget. Procedural is sized once at deployment. The composition itself is what advisors do.

§ 05

Episodic Memory

Episodic memory is the agent's experience log — the verbatim sequence of turns that happened, optionally enriched with tool calls and structured outcomes. It answers questions of the form “what did the user say earlier” and “what did I tell them last time.” In Spring AI it is the responsibility of two collaborating abstractions: ChatMemory (the strategy — what to keep, when to evict) and ChatMemoryRepository (the storage backend).

The core abstractions

ChatMemoryRepository has four methods that constitute the storage contract:

public interface ChatMemoryRepository {
    List<String> findConversationIds();
    List<Message> findByConversationId(String conversationId);
    void saveAll(String conversationId, List<Message> messages);
    void deleteByConversationId(String conversationId);
}

Implementations shipped with Spring AI 1.1.x: InMemoryChatMemoryRepository, JdbcChatMemoryRepository, CassandraChatMemoryRepository, Neo4jChatMemoryRepository, MongoChatMemoryRepository, CosmosDBChatMemoryRepository, plus a Redis implementation via spring-ai-model-chat-memory-repository-redis. The standardized artifact pattern is spring-ai-model-chat-memory-repository-<backend>.

ChatMemory sits above the repository and decides which messages survive eviction. The shipping strategy is MessageWindowChatMemory (sliding window of N messages). The architecture allows custom strategies — token-budgeted, time-based, importance-scored — but the windowed strategy covers most agent use cases.

Recommended production configuration

For our environment — existing PostgreSQL footprint, SOX retention requirements, microservice-per-agent topology — JDBC over PostgreSQL is the default. Cassandra is the alternative when the agent has very high message throughput and natural time-series query patterns (Cassandra's TTL feature is particularly clean for fixed-retention regimes).

@Configuration
public class EpisodicMemoryConfig {

    @Bean
    public ChatMemoryRepository chatMemoryRepository(JdbcTemplate jdbcTemplate) {
        return JdbcChatMemoryRepository.builder()
            .jdbcTemplate(jdbcTemplate)
            .dialect(new PostgresChatMemoryRepositoryDialect())
            .build();
    }

    @Bean
    public ChatMemory episodicMemory(ChatMemoryRepository repository) {
        return MessageWindowChatMemory.builder()
            .chatMemoryRepository(repository)
            .maxMessages(20)
            .build();
    }
}
Schema

Spring AI ships SQL schema scripts at classpath:org/springframework/ai/chat/memory/repository/jdbc/schema-postgresql.sql. Use Flyway or Liquibase to apply it — do not rely on auto-creation in production. The schema is intentionally minimal: (conversation_id, message_type, content, timestamp) indexed on conversation_id.

Conversation ID is the join key

Every episodic operation is keyed by a conversationId. This is where multi-tenancy and identity meet memory. A few standing rules:

  • Never use a raw user ID as the conversation ID — one user has many conversations. Use a UUID per logical session.
  • Pair the conversation ID with a user ID in metadata if your repository supports it (or in a separate conversation_metadata table) so queries like “all sessions for user U” remain possible.
  • For agent-to-agent workflows, the conversation ID identifies the workflow instance, not the originating user. Identity travels separately, via the ToolContext pattern from the identity document.

When using the ChatClient API, the conversation ID is passed as an advisor parameter:

String reply = chatClient.prompt()
    .user(message)
    .advisors(spec -> spec.param(ChatMemory.CONVERSATION_ID, conversationId))
    .call()
    .content();

When driving the memory directly (outside the ChatClient, see §09), the same key is passed to ChatMemory.get(conversationId) and ChatMemory.add(conversationId, messages).

§ 06

Semantic Memory

Semantic memory is the agent's world knowledge — facts, definitions, policies, prior consolidated insights — stored in a form that supports retrieval by meaning rather than by exact key. In Spring AI this is the VectorStore abstraction, backed by an EmbeddingModel that turns text into vectors.

The core abstractions

A VectorStore implements three operations: add(List<Document>), delete(List<String> ids), and similaritySearch(SearchRequest). Documents carry text plus arbitrary metadata; the metadata is what enables filtered retrieval (per-tenant, per-user, per-document-type).

@Configuration
public class SemanticMemoryConfig {

    @Bean
    public VectorStore semanticStore(JdbcTemplate jdbcTemplate,
                                     EmbeddingModel embeddingModel) {
        return PgVectorStore.builder(jdbcTemplate, embeddingModel)
            .dimensions(1024)                                  // matches NIM embedding model
            .distanceType(PgVectorStore.PgDistanceType.COSINE_DISTANCE)
            .indexType(PgVectorStore.PgIndexType.HNSW)
            .initializeSchema(false)                           // managed via Flyway
            .schemaName("ai")
            .vectorTableName("semantic_memory")
            .build();
    }
}
Why PGVector for our environment

The PostgreSQL footprint is already operational. Adding pgvector is a single extension, not a new datastore class to monitor, back up, and patch. Hybrid search (vector + lexical via tsvector) is native. The same database hosts the application's relational state, removing one cross-store consistency problem. Performance is sufficient up to ~10M vectors per index with HNSW; beyond that, dedicated vector databases earn their cost.

Document model and metadata discipline

Every document stored should carry, at minimum, four metadata fields. This discipline is the difference between a vector store that supports governance and one that doesn't.

Metadata fieldPurposeExample
tenant_idMulti-tenancy isolationfhlb-des-moines
scopeVisibility classshared | user:<id> | agent:<id>
sourceProvenance for auditpolicy-doc-v3.2
created_atTemporal filtering, retentionISO-8601 timestamp
 
Document doc = Document.builder()
    .text(chunkText)
    .metadata(Map.of(
        "tenant_id", tenantId,
        "scope", "shared",
        "source", sourceDocumentId,
        "created_at", Instant.now().toString(),
        "doc_type", "policy"
    ))
    .build();
vectorStore.add(List.of(doc));

Retrieval with filters

Every similarity search should specify a filter expression. Unfiltered retrieval against a shared vector store is a multi-tenancy bug waiting to surface.

SearchRequest req = SearchRequest.builder()
    .query(userQuestion)
    .topK(5)
    .similarityThreshold(0.7)
    .filterExpression("tenant_id == 'fhlb-des-moines' && " +
                      "(scope == 'shared' || scope == 'user:" + userId + "')")
    .build();

List<Document> context = vectorStore.similaritySearch(req);
The filter is mandatory, not advisory

A request without a filterExpression will happily return documents from other tenants, other users, and other agents. The default should be a service-layer wrapper that injects tenant and user filters automatically — direct VectorStore access without the wrapper should be allowed only for indexing operations, not for retrieval in user-facing paths.

Embedding model coupling

Vectors are model-specific. A document embedded with nv-embedqa-e5-v5 cannot be meaningfully retrieved against a query embedded with text-embedding-3-small. This has two architectural implications:

  • The vector store is bound to one embedding model at a time. Changing models requires re-embedding everything. Treat the embedding model choice as a long-term commitment.
  • Embedding dimensions must match the table schema. Plan the dimension at table creation; changing it requires a new table. Record the model identifier in metadata so future audits can confirm consistency.
§ 07

Procedural Memory

Procedural memory is how the agent acts — its operational policies and its callable skills. CoALA explicitly distinguishes this from semantic memory: semantic stores what is known, procedural stores how to behave. The distinction matters because the failure modes differ. A stale semantic fact produces a wrong answer. A stale procedural rule produces a wrong action — a more expensive class of mistake.

In Spring AI, procedural memory has two surfaces:

  • The system prompt — the agent's standing instructions, role, and policy boundaries. Loaded before every inference.
  • The tool catalog — @Tool-annotated methods (or programmatic FunctionToolCallback registrations) that the LLM can invoke. The set of available tools is itself part of the agent's behavior policy.

System prompts as versioned configuration

The system prompt is the most consequential single artifact in the agent. Treat it like code, not like a string literal. Store it in Git, version it, review changes through the same PR process as application code, and load it as a resource at startup.

@Configuration
public class ProceduralMemoryConfig {

    @Bean
    public PromptTemplate agentSystemPrompt(
            @Value("classpath:prompts/claims-agent-v3.st") Resource resource)
            throws IOException {
        String template = resource.getContentAsString(StandardCharsets.UTF_8);
        return PromptTemplate.builder().template(template).build();
    }
}

The .st file is a plain text template under version control. The filename's version suffix (-v3) is the deliberately blunt way to track which version produced which behavior — a non-obvious requirement at audit time.

Tools as procedural skills

Each tool is a discrete capability the agent can deploy. The set of available tools shapes the agent's action space — adding or removing a tool changes the agent's behavior more than most system-prompt tweaks. Therefore: the tool registry is part of procedural memory, and changes to it deserve the same governance as system-prompt changes.

@Component
public class AccountTools {

    private final OAuth2AuthorizedClientManager authorizedClientManager;
    private final WebClient accountsApi;

    @Tool(description = "Get the current balance for one of the member's accounts. " +
                       "Use only when the user explicitly asks about a specific account.")
    public AccountBalance getBalance(
            @ToolParam(description = "Account identifier") String accountId,
            ToolContext ctx) {

        Authentication principal = (Authentication) ctx.getContext().get("authentication");
        // OBO exchange + downstream call (see Identity Management document, §07)
        OAuth2AuthorizedClient authorized = authorizedClientManager.authorize(
            OAuth2AuthorizeRequest
                .withClientRegistrationId("obo-accounts")
                .principal(principal)
                .build());

        return accountsApi.get()
            .uri("/accounts/{id}/balance", accountId)
            .header(HttpHeaders.AUTHORIZATION,
                    "Bearer " + authorized.getAccessToken().getTokenValue())
            .retrieve()
            .bodyToMono(AccountBalance.class)
            .block();
    }
}
Tool description is part of procedural memory

The description string on @Tool is what the LLM uses to decide whether to invoke the tool. It is not documentation for humans; it is operational behavior. Changes to a tool description are behavior changes — treat them with the same scrutiny as code changes. Tests should pin the description string so a casual edit doesn't silently shift the agent's tool-selection patterns.

Tool catalog governance

At 50–250 agents, the tool catalog is no longer something each agent manages locally. Tools become a platform asset: registered centrally, discovered via Backstage, governed for who can call what, and audited per invocation. The pattern is:

  • In-process tools (@Tool methods) for tightly coupled capabilities specific to one agent.
  • Remote MCP-hosted tools (per the identity document, §08) for capabilities shared across multiple agents, where governance, separate deployment cadence, and centralized auditability matter.

A capability that ends up needed by three or more agents should migrate from in-process to MCP. The hop cost is real but worth it for centralization.

§ 08

Memory Operations and Lifecycle

Each memory type has the same five operations, but with different mechanics, costs, and failure modes for each. Knowing which operation to optimize for is the difference between a memory system that scales and one that does not.

OperationWorkingEpisodicSemanticProcedural
Read n/a (constructed) Latest N by conv. ID Vector similarity + filter File load at startup
Write n/a (transient) Append on each turn Indexed write (ingestion) PR + redeploy
Update n/a Rare; usually append-only Re-embed and re-index New version of prompt/tool
Forget n/a TTL or explicit delete Delete by ID; tombstone Remove from registry
Consolidate n/a Summarize older turns Extract from episodic Promote learnings to rules

The async-write principle

Episodic writes happen on the user's critical path — they extend the latency of every agent turn. Semantic writes (ingestion) and consolidation (episodic → semantic) should never happen on that path. The pattern:

  • Episodic append is synchronous, single insert, indexed on conversation ID — fast enough to absorb into the turn.
  • Semantic ingestion runs in a background workflow (Kestra) — chunk → embed → metadata enrich → batch insert.
  • Consolidation runs on a schedule (nightly for most agents, hourly for high-throughput ones) over a closed conversation window.

Forgetting is a first-class operation

Memory grows without bound unless explicitly trimmed. In a regulated environment, “just keep everything” is also wrong — retention policies cap how long certain data can live. The defaults:

  • Episodic: TTL at the row level (Cassandra's native TTL; for JDBC, a scheduled DELETE WHERE timestamp < now() - INTERVAL '90 days'). Default 90 days, override per agent class.
  • Semantic: Soft-delete with tombstone metadata, hard-delete on a separate schedule. Re-indexing should respect tombstones.
  • Procedural: Old prompt versions and deprecated tools should be archived, not silently removed — audit trails reference them.

An agent without a forgetting strategy has no retention policy. An agent without a retention policy has no compliance posture. Decide this at design time, not when legal asks.

§ 09

Driving Memory Outside the ChatClient

The ChatClient API is convenient for HTTP-driven, user-facing chat. It is the wrong shape for half of what production agents actually do: scheduled batch jobs, event-driven processors, multi-step workflows orchestrated by Kestra, and any service-layer code that needs precise control over the prompt, the tool execution loop, or the memory write semantics. For these cases, drive the memory primitives directly.

The three layers below ChatClient

From bottom up: ChatModel (one call to the LLM), ToolCallingManager (executes tool calls the model requested), and the memory primitives ChatMemory and VectorStore. ChatClient is sugar around these. The sugar is removable.

@Service
public class BatchAgentService {

    private final ChatModel chatModel;
    private final ChatMemory episodicMemory;
    private final VectorStore semanticStore;
    private final ToolCallingManager toolCallingManager;
    private final List<ToolCallback> toolCallbacks;
    private final PromptTemplate systemPromptTemplate;

    public String runTurn(String conversationId, String userInput, String tenantId) {

        // 1. Procedural — load system prompt
        Message systemMessage = systemPromptTemplate.createMessage();

        // 2. Semantic — retrieve relevant context
        SearchRequest req = SearchRequest.builder()
            .query(userInput)
            .topK(5)
            .similarityThreshold(0.7)
            .filterExpression("tenant_id == '" + tenantId + "'")
            .build();
        String contextBlock = semanticStore.similaritySearch(req).stream()
            .map(Document::getText)
            .collect(Collectors.joining("\n---\n"));

        // 3. Episodic — load history
        List<Message> history = episodicMemory.get(conversationId);

        // 4. Working — compose
        List<Message> messages = new ArrayList<>();
        messages.add(systemMessage);
        if (!contextBlock.isBlank()) {
            messages.add(new SystemMessage("Relevant context:\n" + contextBlock));
        }
        messages.addAll(history);
        UserMessage current = new UserMessage(userInput);
        messages.add(current);

        // 5. Configure with tools, externally-managed execution
        ChatOptions options = ToolCallingChatOptions.builder()
            .toolCallbacks(toolCallbacks)
            .internalToolExecutionEnabled(false)
            .build();

        Prompt prompt = new Prompt(messages, options);

        // 6. Agent loop with explicit tool execution
        ChatResponse response = chatModel.call(prompt);
        while (response.hasToolCalls()) {
            ToolExecutionResult result =
                toolCallingManager.executeToolCalls(prompt, response);
            prompt = new Prompt(result.conversationHistory(), options);
            response = chatModel.call(prompt);
        }

        String reply = response.getResult().getOutput().getText();

        // 7. Episodic — write back
        episodicMemory.add(conversationId, List.of(current, new AssistantMessage(reply)));

        return reply;
    }
}

This is the same agent loop the ChatClient runs internally, expressed directly. Every step is observable, debuggable, and customizable. In return for the verbosity, you get: explicit control over the system message, the ability to inject custom context blocks, full visibility into the tool-execution loop, and the option to short-circuit, batch, or retry tool calls based on application logic.

When to drop down to this level

  • Kestra-orchestrated workflows. Each step is a discrete agent turn, possibly in a different process, with deterministic resumability requirements.
  • Scheduled batch jobs. “Run agent X over the closed claims from yesterday and update the case notes.” No HTTP request; no live user.
  • Event-driven processors. Agent reacts to a pgmq message or a Kafka event. Memory keyed by entity ID, not conversation ID.
  • Custom approval flows. Pause before tool execution, gate it on a human approval, then resume.
  • Multi-model routing. Pick a different ChatModel based on tenant or task complexity. ChatClient wraps one model; the loop above can switch.
Set internalToolExecutionEnabled=false deliberately

By default, ChatModel implementations execute tool calls transparently and loop until done. Setting internalToolExecutionEnabled(false) on ToolCallingChatOptions hands the loop to you. This is required for any approval gating, custom retry logic, or persistence of intermediate tool results. It is also where most bugs creep in — forgetting to loop, mishandling ToolExecutionResult, or losing the conversation history between iterations. Test the loop carefully.

Memory primitives are usable standalone

The episodic and semantic primitives don't need a model at all. They can be used as plain data services from any code in the application:

// Episodic — pure storage operations
List<Message> history = episodicMemory.get(sessionId);
episodicMemory.add(sessionId, List.of(new UserMessage("Hello")));
episodicMemory.clear(sessionId);

// Semantic — ingestion without an agent
List<Document> chunks = new TokenTextSplitter().apply(
    new TikaDocumentReader(resource).get());
chunks.forEach(d -> d.getMetadata().put("tenant_id", tenantId));
semanticStore.add(chunks);

// Semantic — retrieval as a service for any consumer (not just LLMs)
List<Document> matches = semanticStore.similaritySearch(req);

This matters: semantic memory is also a useful enterprise search asset, and episodic memory is also a conversation history endpoint for support UIs. The fact that an LLM consumes them does not mean only an LLM can.

§ 10

Scoping, Tenancy, and Identity

Memory without scoping is a data leak. Three orthogonal scope dimensions need to be resolved for every memory operation: tenant, principal, and agent.

DimensionWhat it isolatesEpisodicSemantic
Tenant Bank / business unit Separate conversation namespace or DB schema tenant_id metadata filter (mandatory)
Principal End user or service Conversation ID is per-session per-user scope = user:<id> for private; shared for org-wide
Agent Which agent “owns” the memory Separate conversation IDs per agent scope = agent:<id> for agent-specific knowledge

Default visibility hierarchy

For any retrieval, the agent should see, in priority order:

  1. User-private memory — facts and preferences the user has shared with this agent (or with this user across all agents).
  2. Agent-shared memory — knowledge specific to this agent's role (e.g., claims agent's procedural knowledge).
  3. Organization-shared memory — bank-wide policies, terminology, public knowledge.

The filter expression encodes this as a boolean composition:

String filter = String.format(
    "tenant_id == '%s' && " +
    "(scope == 'shared' || " +
    " scope == 'user:%s' || " +
    " scope == 'agent:%s')",
    tenantId, userId, agentId);

SearchRequest req = SearchRequest.builder()
    .query(question)
    .topK(8)
    .similarityThreshold(0.7)
    .filterExpression(filter)
    .build();

Identity propagation

The identity flowing through the application (see Identity Management document, §07) is the same identity that scopes memory. The ToolContext pattern for tools applies equally to memory operations:

@Service
public class TenantScopedMemoryService {

    private final VectorStore vectorStore;

    public List<Document> retrieve(String query, Authentication auth) {
        Jwt jwt = (Jwt) auth.getPrincipal();
        String tenantId = jwt.getClaimAsString("tenant_id");
        String userId   = jwt.getSubject();
        if (tenantId == null) {
            throw new AccessDeniedException("Token missing tenant claim");
        }

        SearchRequest req = SearchRequest.builder()
            .query(query)
            .topK(5)
            .similarityThreshold(0.7)
            .filterExpression(scopedFilter(tenantId, userId))
            .build();

        return vectorStore.similaritySearch(req);
    }

    private String scopedFilter(String tenantId, String userId) {
        return "tenant_id == '" + tenantId + "' && " +
               "(scope == 'shared' || scope == 'user:" + userId + "')";
    }
}
Filter injection is the new SQL injection

The filter expression is parsed and applied as a query. String-concatenating untrusted input into it has the same class of risks as string-concatenating SQL. Either validate tenant/user IDs against a strict pattern before composition ([a-zA-Z0-9\-]+ only), or use the framework's structured filter builder when available. Never accept a raw filter expression from outside the trust boundary.

§ 11

Retrieval Strategy and Hybrid Search

Pure vector similarity is the simplest retrieval strategy and the most common reason agents return confidently irrelevant context. Production retrieval combines multiple signals.

The recency / relevance / salience triple

The Generative Agents paper (Park et al., 2023) introduced a useful retrieval scoring composition for episodic memory that has since become a community baseline:

  • Relevance — vector similarity between the query and the candidate memory.
  • Recency — exponential decay on the candidate's age.
  • Salience — how “important” the memory was when stored (scored at write time, often by the LLM).

The final score is a weighted sum; the weights are tuned per use case. This composition is not native to Spring AI but is straightforward to implement on top of VectorStore.similaritySearch by attaching created_at and salience metadata, retrieving a wider candidate set, and re-ranking in application code.

Hybrid search — vector + lexical

Vector search returns documents that are semantically close. It famously misses exact identifiers — an agent looking up “order #A47-2291” will get documents about orders generally rather than that specific one. The mitigation is hybrid search: run a vector query and a keyword query in parallel, then fuse the rankings.

Reciprocal Rank Fusion (RRF) is the standard fuser. Each document gets a score of sum(1 / (k + rank_i)) across the rankers. PGVector pairs naturally with PostgreSQL's tsvector full-text search; the fusion is application-level:

public List<Document> hybridSearch(String query, String tenantId) {

    // 1. Semantic ranking
    List<Document> vectorHits = vectorStore.similaritySearch(
        SearchRequest.builder()
            .query(query).topK(20)
            .filterExpression("tenant_id == '" + tenantId + "'")
            .build());

    // 2. Lexical ranking (parallel, via jdbcTemplate or repository)
    List<Document> lexicalHits = lexicalSearch(query, tenantId, 20);

    // 3. RRF fusion
    return reciprocalRankFusion(vectorHits, lexicalHits, 60).stream()
        .limit(5)
        .toList();
}

private List<Document> reciprocalRankFusion(
        List<Document> a, List<Document> b, int k) {
    Map<String, Double> scores = new HashMap<>();
    rank(a, k, scores);
    rank(b, k, scores);
    return scores.entrySet().stream()
        .sorted(Map.Entry.<String, Double>comparingByValue().reversed())
        .map(e -> findById(a, b, e.getKey()))
        .toList();
}
When hybrid earns its complexity

Pure vector retrieval is sufficient when the corpus is conceptual and the queries are conceptual (“explain our overdraft policy”). Hybrid pays off as soon as queries reference specific entities — IDs, account numbers, dates, product SKUs. For financial workloads where exact reference matters, hybrid is the default.

Re-ranking with a cross-encoder

For high-stakes retrieval, an additional re-ranking step using a cross-encoder model (e.g., a small dedicated re-ranker, or the LLM itself in a structured-output mode) further sharpens the top-K. This adds a network round trip but lifts precision substantially. Worth measuring; not always worth deploying.

§ 12

Memory Consolidation

Episodic memory is verbose — every turn, including small talk and clarifications. Semantic memory is condensed — facts worth remembering across sessions. The bridge is consolidation: a background process that reads recent episodic memory, extracts the durable knowledge, and writes it to semantic memory as a derived artifact.

This is also where human memory and AI memory diverge most usefully. Humans consolidate during sleep; agents consolidate on a Kestra schedule. The pattern is the same.

The consolidation loop

Episodic → Semantic Consolidation
1
Scheduler
Kestra workflow runs nightly. Selects closed conversations from the prior 24 hours (or the agent's configured window).
2
Extractor
For each conversation, an LLM call with a structured-output schema (e.g., BeanOutputConverter<ExtractedFacts>) extracts candidate facts: preferences stated, decisions made, entities mentioned with attributes.
3
Deduplicator
Each candidate is embedded and compared to existing semantic memory. Near-duplicates are merged with an updated timestamp; conflicts are flagged for resolution.
4
Writer
Surviving facts are written to semantic_memory with metadata: source=consolidation, derived_from=<conv_id>, scope=user:<id> or shared, confidence=<score>.
5
Auditor
A digest of new/updated/conflicted facts is written to the audit sink, including the source conversation IDs, for review and recall.

In-line consolidation via custom advisor

For agents where consolidation needs to happen in near-real-time rather than nightly, a custom CallAdvisor can run extraction inline after each turn. The pattern follows the Redis example in the Spring AI community blogs — an advisor with Ordered.HIGHEST_PRECEDENCE + 60 that runs after the main chain, extracts memories from the response, and writes them asynchronously.

@Component
public class ConsolidationAdvisor implements CallAdvisor, Ordered {

    private final ChatModel chatModel;
    private final VectorStore semanticStore;
    private final BeanOutputConverter<ExtractedFacts> converter =
        new BeanOutputConverter<>(ExtractedFacts.class);
    private final TaskExecutor backgroundExecutor;

    @Override public int getOrder() { return Ordered.HIGHEST_PRECEDENCE + 60; }
    @Override public String getName() { return "ConsolidationAdvisor"; }

    @Override
    public ChatClientResponse adviseCall(ChatClientRequest req, CallAdvisorChain chain) {
        ChatClientResponse response = chain.nextCall(req);

        // Extract on a background thread — do not extend the turn's latency
        backgroundExecutor.execute(() -> extractAndStore(req, response));

        return response;
    }

    private void extractAndStore(ChatClientRequest req, ChatClientResponse response) {
        String prompt = """
            Extract durable facts about the user from this exchange.
            Return JSON: %s
            User: %s
            Assistant: %s
            """.formatted(converter.getFormat(),
                          extractUserText(req), extractAssistantText(response));
        try {
            ChatResponse extraction = chatModel.call(new Prompt(prompt));
            ExtractedFacts facts = converter.convert(
                extraction.getResult().getOutput().getText());
            persist(facts);
        } catch (Exception e) {
            // Consolidation failures must not break the user-facing turn
            log.warn("Consolidation failed", e);
        }
    }
}
Consolidation must be idempotent and reversible

A consolidation bug that writes wrong facts is worse than not consolidating at all — bad semantic memory poisons every future retrieval. Always tag derived memories with derived_from pointing back to the source conversation, and provide a tool to revoke all derived memories from a given source. Without this, errors compound and become unrecoverable.

§ 13

Governance, Audit, and Forgetting

In a SOX environment, memory is regulated data. The same scrutiny that applies to a database of customer records applies to the agent's episodic store and semantic store. The architectural requirements have four pillars: retention, auditability, access control, and right to be forgotten.

Retention

Every memory store must have a declared retention policy enforced by automation, not by good intentions:

StoreDefault retentionMechanism
Episodic (JDBC / PostgreSQL)90 daysScheduled DELETE job, partitioned tables
Episodic (Cassandra)90 daysNative row-level TTL
Semantic (user-scoped)Lifetime of user + 90 days post-deletionMetadata flag + sweep job
Semantic (shared)Indefinite, version-controlledIngestion provenance tracking
Procedural (prompts/tools)Git history retained indefinitelyStandard SCM
Consolidation audit log7 years (SOX-aligned)Immutable append-only sink

Auditability

Every memory write should be observable. The mechanism is the same as for any other production action: structured logs with trace ID + identity context. The minimum schema for a memory-write audit record:

{
  "event": "memory.write",
  "memory_type": "semantic",
  "operation": "add",
  "tenant_id": "fhlb-des-moines",
  "principal": { "sub": "user-12345", "act": "agent-claims" },
  "scope": "user:user-12345",
  "source": "consolidation",
  "derived_from": "conv-abc-123",
  "document_ids": ["doc-xyz-789"],
  "trace_id": "01HQ8...",
  "timestamp": "2026-05-27T14:32:11Z"
}

This record joins the standard application audit stream. Trace ID is the join key against operational observability (Phoenix / Langfuse). For high-value memories (e.g., agent-generated decisions that influenced a downstream action), forward a copy to the dedicated audit sink discussed in the identity document, §11.

Right to be forgotten

GDPR and similar regulations require the ability to delete all data associated with a specific user on request. For memory this means a coordinated operation across all three persistent stores:

@Service
public class MemoryDeletionService {

    private final ChatMemoryRepository episodicRepo;
    private final VectorStore semanticStore;
    private final JdbcTemplate jdbcTemplate;
    private final AuditLogger audit;

    @Transactional
    public DeletionReport forgetUser(String tenantId, String userId, String requestId) {
        audit.log("memory.delete.start",
            Map.of("user", userId, "request", requestId));

        // 1. Episodic — delete all conversations for the user
        List<String> conversationIds = jdbcTemplate.queryForList(
            "SELECT id FROM conversation_metadata WHERE user_id = ? AND tenant_id = ?",
            String.class, userId, tenantId);
        conversationIds.forEach(episodicRepo::deleteByConversationId);

        // 2. Semantic — delete user-scoped documents and consolidation derivatives
        // (Implementation depends on VectorStore; PGVector supports SQL DELETE)
        int semanticDeleted = jdbcTemplate.update(
            "DELETE FROM ai.semantic_memory " +
            "WHERE metadata->>'tenant_id' = ? " +
            "  AND (metadata->>'scope' = ? OR " +
            "       metadata->>'derived_from' IN (SELECT id FROM conversation_metadata " +
            "                                      WHERE user_id = ?))",
            tenantId, "user:" + userId, userId);

        // 3. Audit — record the deletion itself (this record is retained)
        DeletionReport report = new DeletionReport(
            conversationIds.size(), semanticDeleted, Instant.now());
        audit.log("memory.delete.complete",
            Map.of("user", userId, "request", requestId, "report", report));

        return report;
    }
}
The deletion record is retained — that is the point

Deleting a user's data does not mean forgetting that the deletion happened. The audit record of the deletion is preserved indefinitely, with the request ID linking back to the legal/compliance request that triggered it. This is the standard pattern across GDPR-compliant systems and applies cleanly here.

Memory poisoning as an attack class

Agents that write to their own memory create a new attack surface: memory poisoning. An attacker who can influence what the agent sees in one session — by injecting content into a document, an email, or a prior conversation — can plant misleading “facts” that get consolidated into long-term memory and influence future sessions, including those involving other users.

Mitigations:

  • Provenance metadata on every semantic write. Always know the source — user input, document ingestion, consolidation. Treat sources differently at retrieval time.
  • Trust tiers on retrieval. Memories sourced from authoritative documents weight higher than memories derived from conversation.
  • Consolidation must not auto-promote user-asserted facts to shared scope. A fact a user tells the agent stays in user:<id> scope unless explicitly curated into shared.
  • Approval gates for high-value memory writes. Spring AI's emerging ToolApprovalStrategy hook generalizes to memory writes; require human or rule-based approval for writes that could affect shared knowledge.
§ 14

Failure Modes

The architectural patterns above prevent specific failures. Knowing the failures by name makes the patterns easier to defend in review.

Failure modeSymptomMitigation
Context window overflow Old turns silently truncated; agent forgets earlier instructions Explicit token budgets per plane; summarization for older turns
Cross-tenant leak Retrieval returns documents from another tenant Mandatory tenant_id filter wrapper; integration tests verify isolation
Stale procedural drift Tool descriptions or system prompts edited without review; agent behavior changes silently System prompt + tool descriptions in Git, change review required
Embedding/model mismatch Retrieval quality degrades after embedding model change Re-embed everything; record embedding model in metadata
Consolidation poisoning Wrong facts accumulate; retrieval surfaces hallucinations Provenance tracking; reversible consolidation; trust tiers
Episodic store unbounded growth Database fills; latency degrades TTL or scheduled deletion; partitioned tables
Stuck conversation IDs Same conversation ID reused across sessions; memory bleeds UUID per session; never derive from user ID alone
Hot top-K Same few documents always retrieved regardless of query Similarity threshold + diversity re-ranking; tune topK
Memory-as-conversation history exposure Users see other users' conversations via a debug endpoint Conversation ID lookup endpoints require IDOR-safe authorization

Every memory bug a production agent ships with is a bug it ships to every conversation. Test memory in isolation, with synthetic data, before any user-facing rollout.

§ 15

Patterns to Establish

These are the principles worth codifying as platform-wide standards, applicable to every new agent and to any non-AI service that maintains stateful context.

  1. Four memory types, named explicitly. Working, episodic, semantic, procedural. Every architecture review for a new agent answers: what does each plane look like for this agent?
  2. Episodic via ChatMemoryRepository, persisted by default. JDBC over PostgreSQL is the standard; Cassandra for high-throughput. In-memory is for local development only.
  3. Semantic via VectorStore, PGVector by default. Mandatory metadata: tenant_id, scope, source, created_at. Retrieval through a tenant-scoped service wrapper; never directly.
  4. Procedural in Git, versioned, reviewed. System prompts as resource files. Tool descriptions are behavior — changes require PR review.
  5. One conversation ID = one session. UUID per session, never derived from user ID alone. Conversation metadata table maps conversation → user/tenant/agent for governance queries.
  6. Filter expressions are mandatory on semantic retrieval. Default-deny: code paths that bypass the tenant-scoped wrapper require explicit security review.
  7. Memory primitives are usable outside ChatClient. Batch jobs and Kestra workflows call ChatMemory, VectorStore, and ToolCallingManager directly. The ChatClient sugar is removable when the situation warrants.
  8. Retention policy is declared per agent class. Default 90 days episodic; semantic per scope; consolidation audit log 7 years.
  9. Right to be forgotten is implemented before launch. Coordinated deletion across episodic, semantic, and derivatives. Deletion itself is audited.
  10. Consolidation is async, reversible, and provenance-tagged. Never on the user's critical path. Every derived memory points back to its source. User-asserted facts do not auto-promote to shared scope.
  11. Identity propagates into memory. The same Authentication that scopes downstream API calls scopes memory reads and writes. tenant_id always comes from a validated token claim, never from request input.
  12. Memory writes are observable. Structured audit events on every persistent write, joined by trace ID to operational observability.

The architecture for AI memory is the architecture for any stateful, multi-tenant, regulated service that retains context across sessions. The AI use case forces the conversation; the patterns benefit everything that comes after.

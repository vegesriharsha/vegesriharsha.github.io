Architecture Learning Document

Identity Management in Agentic Systems

A working architect's notes on extending OAuth 2.0 patterns — user identity, workload identity, and delegated identity — across Spring AI agents, MCP servers, and the broader enterprise.

Audience
Platform, Security, and AI engineering teams
Scope
Conceptual foundations + implementation guidance
Status
Living document — patterns evolving
Framework versions assumed

Patterns and code in this document are anchored to: Spring Boot 3.4+ / 4.0, Spring Security 6.3+ / 7.x (native RFC 8693 token exchange), Spring AI 1.1.x (stable) or 2.0.x (M-series). MCP transport assumes Streamable HTTP per MCP spec revision 2025-03-26 (SSE is deprecated). Spring AI MCP Security modules are community-driven (group org.springaicommunity) and currently at version 0.1.x — production-suitable but pre-1.0; track for breaking changes. OpenShift 4.19+ supports direct external OIDC. PingFederate supports RFC 8693 token exchange grant; verify it is enabled on your tenant.

§ 01

Context and Foundations

Agentic systems force a re-examination of identity patterns that traditional request/response APIs let us paper over. A deterministic client calling a deterministic service can authenticate once and forward credentials; the failure modes are well understood. An LLM-driven agent introduces non-determinism in tool selection, multi-hop call chains across trust boundaries, and ambient assumptions about “acting on behalf of” the user that the underlying protocols never made explicit.

The patterns established to solve identity for agents are not narrowly applicable to AI workloads. They are the modern OAuth 2.0 patterns — workload identity federation, token exchange for delegation, audience-bound scopes, full actor chains in audit trails — that every cloud-native service needs once it leaves the comfort of a trusted internal network. The AI use case forces the architectural conversation; the architecture benefits every workload that follows.

The identity patterns we adopt for agents become the identity patterns for everything. Get them right once.

This document captures the architectural reasoning, the implementation primitives available in the Spring ecosystem, and the open design decisions that remain. It is intentionally framed in concepts rather than tied to any specific application, so the team can apply the same reasoning to whichever workload comes next.

§ 02

The Architectural Shift

The traditional service-to-service authentication model — long-lived client credentials issued to a service account, used for the client_credentials grant — is a machine-to-machine pattern. It answers one question: which application is calling? It loses the human user entirely. Downstream services see “app X called” with no notion of which end-user prompted the call.

This model worked when applications were monoliths talking to other monoliths over trusted networks, where the user identity stayed within the originating application's session and never needed to cross service boundaries. It breaks in three distinct ways for modern systems:

  • The user identity matters downstream. Authorization decisions, audit logs, and rate-limiting all depend on knowing “which user is this for,” not just “which app is calling.”
  • Workloads no longer have stable machine identities. Pods on Kubernetes are ephemeral, replicated, and scheduled dynamically. There is no equivalent of the Windows service account tied to a host.
  • An intermediate reasoning layer sits between user and downstream. An LLM (or any orchestration logic) makes decisions that affect what downstream calls happen. Blindly forwarding the user's original token misses the fact that the agent is the actor; using only the agent's own credentials misses the user entirely.

The shift is from one identity plane (the app's machine identity) to three identity planes, each with distinct mechanics and purposes.

§ 03

The Three Identity Planes

Every modern service participates in three identity planes simultaneously. Treating them as one is the root cause of most identity-related architecture problems.

PLANE 01
User Identity
Who is the end-user whose request is being processed? Established at the edge through OIDC. Carried as a JWT access token. Validated by every service as an OAuth 2.0 resource server.
PLANE 02
Workload Identity
Which pod is making this call? Established by the cluster, federated to the IdP. Carried as a short-lived projected ServiceAccount token. Replaces long-lived client secrets entirely.
PLANE 03
Delegated Identity
Workload W is acting on behalf of User U toward downstream D. Established via OAuth 2.0 Token Exchange (RFC 8693). Combines subject (user) and actor (workload) into a single audience-scoped token.

The remainder of this document treats these planes individually before showing how they compose. The mental model to hold: every outbound call from a service must answer all three questions — “who's the user, who's the workload, who's the audience” — and the answer is encoded into the token presented at that call.

§ 04

User Identity at the Service Boundary

Every service that accepts user-bound calls must validate the incoming JWT as an OAuth 2.0 resource server. This is the simplest of the three planes, but the most consequential: it is the moment the rest of the system knows who the user is. If this validation is weak or missing, no amount of downstream sophistication recovers the situation.

What validation must include

  • Signature against the IdP's published JWKS (rotating keys handled by the library)
  • Issuer matches the expected IdP issuer URI exactly
  • Audience includes this service's identifier — a token issued for service A must not validate at service B
  • Expiry and not-before are within acceptable clock skew
  • Required scopes are present for the requested operation
Common pitfall

Skipping audience validation is the single most common configuration error. Without it, any token issued by your IdP — for any service — passes validation at every service. The token becomes a bearer credential good across the entire enterprise. Always validate aud.

Spring Security configuration

The configuration below validates inbound tokens, enforces audience binding, requires a scope for the chat endpoint, and is otherwise stateless.

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Value("${idp.issuer}")
    private String issuer;

    @Value("${idp.audience}")
    private String expectedAudience;

    @Bean
    SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(a -> a
                .requestMatchers("/actuator/health/**").permitAll()
                .requestMatchers("/api/chat/**").hasAuthority("SCOPE_chat:invoke")
                .anyRequest().authenticated())
            .oauth2ResourceServer(o -> o.jwt(Customizer.withDefaults()))
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS));
        return http.build();
    }

    @Bean
    JwtDecoder jwtDecoder() {
        NimbusJwtDecoder decoder = JwtDecoders.fromIssuerLocation(issuer);
        OAuth2TokenValidator<Jwt> validators = new DelegatingOAuth2TokenValidator<>(
            JwtValidators.createDefaultWithIssuer(issuer),
            new JwtClaimValidator<List<String>>("aud", aud -> aud.contains(expectedAudience))
        );
        decoder.setJwtValidator(validators);
        return decoder;
    }
}

Note that the bulk of validation is handled by the framework — the manual addition is the audience check, which Spring Security does not enforce by default and which is non-negotiable.

§ 05

Workload Identity in OpenShift

The replacement for Windows service accounts (or any pattern that stores a long-lived secret in the workload) is workload identity federation. The cluster mints a short-lived, cryptographically signed identity token for each pod, and the IdP is configured to trust the cluster as a federated identity source.

The mechanism: a projected ServiceAccount token. Kubernetes mounts a JWT into the pod's filesystem, signed by the cluster's own signing key, with the IdP as the audience. The kubelet auto-rotates the file before expiry. The pod presents this JWT to the IdP, which validates it against the cluster's OIDC discovery endpoint and issues a real access token in return.

Workload Identity Federation
1
Kubelet
Mints a JWT signed by the cluster, with the IdP as audience. Mounted at /var/run/secrets/<path>/token. Auto-rotated at ~80% of expiry.
2
Pod
Reads the projected token. Presents it to the IdP as a JWT bearer assertion or as part of a token exchange.
3
IdP
Validates the JWT against the cluster's OIDC discovery endpoint (configured as a trusted issuer). Issues a real access token bound to the pod's workload identity.
4
Pod
Uses the access token for outbound calls. No secret ever lives on disk.

Deployment manifest

spec:
  serviceAccountName: claims-agent-sa
  containers:
    - name: agent
      volumeMounts:
        - name: idp-id-token
          mountPath: /var/run/secrets/idp
          readOnly: true
  volumes:
    - name: idp-id-token
      projected:
        sources:
          - serviceAccountToken:
              audience: https://idp.internal
              expirationSeconds: 3600
              path: token

Token loader with rotation awareness

@Component
public class WorkloadTokenProvider {

    private static final Path TOKEN_PATH = Path.of("/var/run/secrets/idp/token");
    private final AtomicReference<CachedToken> cache = new AtomicReference<>();

    public String getToken() {
        CachedToken current = cache.get();
        if (current == null || current.isStale()) {
            try {
                String raw = Files.readString(TOKEN_PATH).trim();
                cache.set(new CachedToken(raw, Instant.now().plus(Duration.ofMinutes(10))));
                return raw;
            } catch (IOException e) {
                throw new IllegalStateException("Cannot read projected SA token", e);
            }
        }
        return current.token();
    }

    private record CachedToken(String token, Instant refreshAfter) {
        boolean isStale() { return Instant.now().isAfter(refreshAfter); }
    }
}
What this eliminates

No client secret in the pod. No secret to rotate. No credential to leak in environment variables or logs. The pod's identity is its ServiceAccount, cryptographically provable, ephemeral by construction. This is the single largest security improvement available when moving workloads to OpenShift.

The IdP-side configuration

The IdP needs to be configured to:

  • Trust the cluster's OIDC discovery endpoint as a token issuer (requires the JWKS endpoint to be reachable from the IdP)
  • Map ServiceAccount identities to internal client/workload identities, typically via a claim-to-client mapping rule
  • Issue access tokens scoped appropriately for the workload's intended downstream calls

This is one-time platform setup. Once in place, every new workload onboarded to the cluster gets identity for free by declaring a ServiceAccount.

§ 06

Delegated Identity via Token Exchange

When a service needs to call a downstream API on behalf of a user, neither plain token forwarding nor the workload's own credentials are correct:

  • Forwarding the user's token works but loses the audit trail of which workload performed the action, and over-shares the token to a service that may not be its intended audience.
  • Using only the workload's credentials loses the user identity entirely — downstream services see “agent called” but not “agent called for user X.”

The correct mechanism is OAuth 2.0 Token Exchange (RFC 8693). The service presents the IdP with two tokens:

ParameterValueMeaning
subject_tokenUser's access tokenThe party on whose behalf the call is being made
actor_tokenWorkload's identity tokenThe party performing the action
audienceDownstream service URIThe intended recipient of the new token
scopeSubset of user's scopesNarrowed scope for least privilege

The IdP returns a new token with sub set to the user, act set to the workload, audience set to the downstream service, and scopes narrowed to what was requested. The downstream service sees the full delegation chain: this call is for user X, performed by workload Y, intended for me.

Spring Security 6.3+ native support

Spring Security has first-class support for the token-exchange grant type. No custom WebClient, no hand-rolled form posts. Register a client with the token-exchange grant type, and the framework's OAuth2AuthorizedClientManager handles subject token extraction, the exchange call, caching, and refresh.

spring:
  security:
    oauth2:
      client:
        registration:
          obo-accounts:
            provider: idp
            client-id: ${WORKLOAD_CLIENT_ID}
            authorization-grant-type: urn:ietf:params:oauth:grant-type:token-exchange
            scope: account:read
          obo-claims:
            provider: idp
            client-id: ${WORKLOAD_CLIENT_ID}
            authorization-grant-type: urn:ietf:params:oauth:grant-type:token-exchange
            scope: claims:read
        provider:
          idp:
            issuer-uri: ${IDP_ISSUER}

One registration per audience. Each registration is scoped narrowly. This is by design: the cost of an extra YAML block is trivial; the audit and security benefits of audience- and scope-specific exchange registrations are substantial.

@Bean
OAuth2AuthorizedClientManager authorizedClientManager(
        ClientRegistrationRepository clients,
        OAuth2AuthorizedClientRepository authorizedClients) {

    OAuth2AuthorizedClientProvider provider = OAuth2AuthorizedClientProviderBuilder.builder()
        .provider(new TokenExchangeOAuth2AuthorizedClientProvider())
        .build();

    DefaultOAuth2AuthorizedClientManager manager =
        new DefaultOAuth2AuthorizedClientManager(clients, authorizedClients);
    manager.setAuthorizedClientProvider(provider);
    return manager;
}

Performing the exchange

At the point where a downstream call is needed, ask the manager for an authorized client. The framework reads the inbound JWT from the security context as the subject token automatically.

OAuth2AuthorizedClient authorized = authorizedClientManager.authorize(
    OAuth2AuthorizeRequest
        .withClientRegistrationId("obo-accounts")
        .principal(SecurityContextHolder.getContext().getAuthentication())
        .build());

String oboToken = authorized.getAccessToken().getTokenValue();

That is the complete delegation primitive. The exchanged token can now be presented to the downstream service as a bearer credential. The downstream service validates it as a normal resource server and observes the full sub + act chain.

Workload identity as the actor

The client authentication for the exchange request should use the workload's projected ServiceAccount token (typically via a JWT bearer client authentication method), not a client secret. The IdP needs to accept the federated assertion as client authentication. This is what closes the loop: no secrets, end-to-end.

§ 07

Wiring Identity Into Spring AI Agents

Agents introduce a wrinkle: the LLM decides which tools to call. Identity must be available to the tool implementation at the moment of invocation, but the LLM's tool selection happens on the framework's terms, often on different threads, sometimes in reactive flows. SecurityContextHolder alone is not reliable across that boundary.

The solution Spring AI provides is ToolContext — a per-invocation key-value map that travels with the request through the framework, accessible inside the tool method. This is the only clean propagation path.

Controller — capture the validated user

@RestController
@RequestMapping("/api/chat")
class ChatController {

    private final ChatService chatService;

    @PostMapping
    ChatResponse chat(@RequestBody ChatRequest req,
                      JwtAuthenticationToken auth) {
        return chatService.chat(req.message(), auth);
    }
}

Service — propagate the Authentication into ToolContext

@Service
class ChatService {

    private final ChatClient chatClient;
    private final AccountTools accountTools;

    String chat(String message, JwtAuthenticationToken auth) {
        return chatClient.prompt()
            .user(message)
            .toolContext(Map.of("authentication", auth))
            .tools(accountTools)
            .call()
            .content();
    }
}

The full Authentication object is what OAuth2AuthorizedClientManager needs to extract the subject token. Passing only the raw token string forces brittle reconstruction inside the tool.

Tool — perform delegation at invocation

@Component
public class AccountTools {

    private final OAuth2AuthorizedClientManager authorizedClientManager;
    private final WebClient accountsApi;

    @Tool(description = "Get current balance for one of the member's accounts")
    public AccountBalance getBalance(
            @ToolParam(description = "Account identifier") String accountId,
            ToolContext ctx) {

        Authentication principal = (Authentication) ctx.getContext().get("authentication");

        OAuth2AuthorizedClient authorized = authorizedClientManager.authorize(
            OAuth2AuthorizeRequest
                .withClientRegistrationId("obo-accounts")
                .principal(principal)
                .build());

        return accountsApi.get()
            .uri("/accounts/{id}/balance", accountId)
            .header(HttpHeaders.AUTHORIZATION,
                    "Bearer " + authorized.getAccessToken().getTokenValue())
            .retrieve()
            .bodyToMono(AccountBalance.class)
            .block();
    }
}

The tool participates in the same identity flow as any controller-driven call. The downstream accounts-api validates the token like any resource server and sees the full delegation chain in the claims.

Why not SecurityContextHolder

Spring AI may execute tool methods on threads that do not inherit the security context. Reactive flows discard ThreadLocal entirely. ToolContext is explicit, per-invocation, and survives both — it is also better for auditing because identity propagation is visible in code rather than implicit through framework magic.

§ 08

MCP Servers in the Delegation Chain

When tools are hosted in remote MCP servers rather than as in-process @Tool methods, the delegation chain extends by one hop. The pattern is the same: every hop is a resource server plus a token-exchange client. Each hop validates the inbound token, performs an exchange to obtain a token audienced for the next hop, and passes that forward.

The Spring AI ecosystem provides this as first-class infrastructure:

  • MCP Server Security module — auto-configures the MCP server as a Spring Security resource server. Inbound JWTs are validated against the configured issuer, audience, and scopes. No custom auth code in the server.
  • MCP Client Security module — auto-configures OAuth2 client behavior on the MCP transport. The exchanged OBO token is automatically attached as the bearer credential on outbound MCP calls.
  • Spring Security 6.3+ token exchange — performs the OBO exchange between hops with no additional code beyond YAML registration.
Transport: use Streamable HTTP, not SSE

The MCP specification revision 2025-03-26 deprecated the legacy HTTP+SSE transport in favor of Streamable HTTP (single endpoint, supports both synchronous and streaming responses). Spring AI 1.1.7 / 2.0.0-M7 (May 2026) made Streamable HTTP the default server protocol. New MCP servers should use Streamable HTTP exclusively; existing SSE deployments should plan migration. Configure via spring.ai.mcp.server.protocol=STREAMABLE.

Maturity caveat — Spring AI MCP Security

The org.springaicommunity MCP Security modules (server, client, authorization server) are community-driven and currently at version 0.1.x. The patterns are correct and production-suitable, but the API surface may shift before 1.0. Pin versions explicitly and track the release cadence.

End-to-end delegation chain

Full Delegation Chain — User → Agent → MCP → Downstream
1
User Edge
User authenticates via OIDC. Frontend obtains user access token. Forwards as bearer to agent service.
2
Agent Service
Validates as resource server. Exchanges user token → OBO token audienced for MCP server. sub = user, act = agent.
3
MCP Client Transport
Auto-attaches OBO token to outbound MCP call (auto-configured by Spring AI MCP Client Security).
4
MCP Server
Validates inbound OBO token as resource server. Exchanges again → token audienced for downstream API. sub = user, act = [agent → mcp-server] (chained).
5
Downstream API
Validates and sees the full actor chain. Performs authorization. Logs the complete provenance.

MCP server resource-server configuration

spring:
  ai:
    mcp:
      server:
        protocol: STREAMABLE   # Streamable HTTP — supersedes deprecated SSE
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: ${IDP_ISSUER}
      client:
        registration:
          obo-downstream:
            provider: idp
            client-id: ${MCP_WORKLOAD_CLIENT_ID}
            authorization-grant-type: urn:ietf:params:oauth:grant-type:token-exchange
            scope: downstream:read
        provider:
          idp:
            issuer-uri: ${IDP_ISSUER}

This is the same configuration shape as any other service in the chain. There is no MCP-specific identity code — only standard Spring Security primitives applied to the MCP transport.

The actor chain

At each hop, the new token's act claim contains the previous actor, recursively. By the time the call reaches the downstream API, the JWT contains a complete chain:

{
  "sub": "user-12345",
  "aud": "downstream-api",
  "iss": "https://idp.internal",
  "exp": 1700000000,
  "scope": "downstream:read",
  "act": {
    "sub": "mcp-server-claims",
    "act": {
      "sub": "agent-service"
    }
  }
}

The downstream API can extract and log this chain to produce audit records of the form: action performed for user-12345, by mcp-server-claims, on behalf of agent-service. Full provenance, no custom plumbing.

Transport caveat

The resource-server + token-exchange pattern applies to Streamable HTTP (and any other network-bound MCP transport). Stdio MCP servers run in-process with the agent and inherit its security context directly — there's no network hop, no separate validation needed. If a stdio MCP server makes outbound calls, those follow the same delegation pattern as any in-process tool.

§ 09

Long-Running Agents and Token Expiry

Short-lived tokens are a security feature. They become a design problem when agent execution time exceeds token lifetime. Three cases arise, each with a different appropriate pattern.

Case 1 — Multi-turn conversations

A chat session lasts longer than a single token's lifetime, but the user's browser session remains active. The cleanest pattern: the frontend always sends a fresh token. The backing service holds the refresh token (never exposed to the browser), refreshes the access token before each agent call, and sends the current one.

The agent service never has to think about user-token refresh. Each turn is independent. Conversation history is stored by user identifier (from the token's sub claim), but the token itself is request-scoped and never persisted on the agent side.

Case 2 — Long tool chains within a single turn

An OBO token's lifetime is bounded by the shorter of subject and actor lifetimes (RFC 8693). If the user's token has 4 minutes left when the agent turn begins, every OBO token minted during the turn shares that ceiling. A 90-second turn with 8 sequential tool calls can fail at tool 6 with a 401.

Two mitigations:

  • Front-load token exchange. If the set of likely tools can be inferred from the LLM's first reasoning step, exchange all OBO tokens up front. All share the same expiry ceiling but you avoid mid-chain refresh complexity.
  • Validate subject token expiry before each tool call. If the user's token is within (say) 60 seconds of expiry, fail fast with a specific error code the frontend can recognize, prompting refresh-and-retry.

The deeper fix is making token lifetimes long enough for normal agent turns. 15 minutes is generally fine; 5 minutes isn't. This is an IdP configuration decision worth raising — agent workloads have different lifetime needs than CRUD APIs.

Case 3 — True long-running agents

An agent that runs for hours, with no active user session to refresh against. This is the architecturally hardest case. Three approaches, in increasing order of correctness:

  • Refresh tokens stored server-side. Capture an OIDC refresh token (offline_access scope) at workflow initiation. Encrypt and store it, associated with the workflow ID. Use it to mint fresh access tokens as needed. Functional but creates a high-value credential at rest.
  • Consent-bound delegation grants. User explicitly grants the agent a time-bounded, scope-limited delegation (“may act as me for these scopes, for 24 hours, for workflow X”). Cleaner audit, narrower blast radius, explicit user consent. Requires IdP support for this kind of grant — custom work in most environments.
  • Re-authentication checkpoints. Agent runs as far as it can. When a step requires fresh user identity, it pauses, notifies the user, the user re-authenticates, the workflow resumes. Most secure, worst UX, appropriate for high-value actions.

Token expiry forces explicit answers to a question that otherwise stays implicit: for how long, and under what conditions, is the agent authorized to act as the user? The right answer depends on the action's risk.

Architecturally: long-running agent workflows should be designed with identity-expiry checkpoints, not assumed to run with ambient user identity forever. Token lifetime is a security control.

§ 10

Scope Design for Agents

Agents amplify the importance of scope design because the LLM's tool selection is non-deterministic. A traditional client's behavior is predictable — scopes can be broad because the developers control what the client does. With an agent, the attack surface becomes whatever the LLM can be convinced to do. Broad scopes are dangerous in a way they aren't for deterministic clients.

Capability vs. resource scopes

Two design philosophies, both viable, best combined:

  • Capability scopes describe actions: account:read, transfer:execute, claims:write.
  • Resource scopes describe data domains: accounts, transfers, claims.

Most mature designs combine them into resource:capability form: account:read, account:write, transfer:execute. This composes cleanly across tools and remains readable in audit logs.

Scope minimization at exchange

The single biggest blast-radius reduction lever. When the agent exchanges a user token for an OBO token, it requests only the scopes the specific downstream call needs, not all scopes the user has.

A user token might carry: account:read account:write transfer:execute statements:read.

The OBO token for a balance lookup should carry only: account:read.

RFC 8693 supports this via the scope parameter on the exchange request. In Spring Security, this is the scope attribute on the client registration. Each downstream call gets the smallest credential that suffices.

Downscoping, not upscoping

Token exchange can narrow scopes but never broaden them — the IdP enforces that exchanged scopes are a subset of subject token scopes. This is a security feature: even a compromised agent cannot elevate privilege beyond what the user already has. It shapes how to design the initial user-token scope set.

Sensitive scopes and step-up authentication

Some actions should not be reachable by an agent without fresh user re-authentication, regardless of what's in the token. Mark certain scopes as “step-up required” at the IdP, requiring an acr (Authentication Context Class Reference) claim proving recent strong authentication.

The agent's OBO token will not carry this acr claim unless the user just re-authenticated. Downstream APIs check acr for sensitive operations and reject calls without it. This effectively keeps high-value actions out of the agent's autonomous reach.

Scope vs. authorization policy

Scopes describe what a token is allowed to do. They do not replace authorization policy in the downstream API. A common mistake is conflating them.

LayerQuestionWhere it lives
Scope checkCan this token read accounts?Token claims + Spring Security
Policy checkCan this user read this account?Application logic + data ownership

Both checks must happen. Scope is necessary but never sufficient.

Tool-to-scope mapping

Each tool should declare which scopes it requires. This becomes documentation, audit material, and the basis for the token-exchange request. Encode it as an annotation, a registry, or part of the tool definition — whatever fits the codebase — but make it explicit and reviewable, not inferred at runtime.

The agent gets the smallest credential that does the job, for the shortest time, against the most specific audience. Everything else is a deviation that requires justification.

§ 11

Observability with Identity Context

Most of what would traditionally be called “audit infrastructure” for agents is already provided by the LLM-tracing stack — OpenTelemetry plus an LLM-aware backend like Phoenix. The hard parts of agent observability — correlating calls across an agent turn, capturing the causal structure of LLM reasoning and tool invocation, recording prompts and responses — are solved problems via OpenInference semantic conventions.

The identity-specific work is not to build a parallel audit pipeline. It is to attach identity context to the spans the tracing stack is already producing, and to forward a subset of those attributes to a dedicated audit sink for compliance retention.

What Phoenix/OTel handles natively

Two semantic-convention layers are relevant here, and they overlap but are not identical:

  • OpenTelemetry GenAI semantic conventions — defines gen_ai.* attributes (model, token usage, finish reasons, system/input/output messages). Client spans for LLM calls reached stable status in early 2026. Agent and framework spans remain experimental but are stable in practice.
  • OpenInference semantic conventions — Phoenix's native convention set, extending OTel with LLM-application-specific attributes (span kind, tool parameters, embedding vectors) and providing helper context managers (using_user, using_session, using_metadata, using_tags) that attach identity and session attributes to spans idiomatically.

Phoenix accepts both. Use OpenInference helpers when emitting from instrumented code; use raw OTel attributes when interop with non-Phoenix backends matters.

The free wins:

  • Correlation across the turn. Trace context propagates from inbound HTTP through LLM calls, tool invocations, and downstream API calls. One trace equals one user turn.
  • Causal structure. Parent-child span relationships capture the agent's reasoning tree — prompt → tool selection → tool call → downstream call → next reasoning step.
  • Prompt and response capture. OpenInference conventions standardize how prompts, completions, tool inputs, and tool outputs land on spans.
  • Operational signal. Latency, token usage, cost, errors come along for free via gen_ai.*.

Identity attributes to attach

The work is in span attributes. At each layer, add identity context:

Span layerAttributes
Agent entry user.id (current OTel direction) or enduser.id (legacy but still widely supported); session.id; OpenInference using_user(id) context manager when Python-instrumenting Phoenix directly
Token exchange auth.subject, auth.actor, auth.audience, auth.scopes_requested, auth.scopes_granted
Tool / downstream call auth.actor_chain, auth.token_audience
User identity attribute is sensitive

The OTel spec recommends that SDKs drop enduser.* attributes by default for PII reasons, with an explicit opt-in to retain them. Verify the SDK's behavior in your environment and treat retention as a deliberate decision — not a default. The newer user.id attribute is positioning as the canonical authenticated-user identifier; enduser.id remains in the spec but its deprecation status is in flux. Pick one, document the choice, and apply it consistently — querying across two attribute names later is painful.

With these attributes in place, Phoenix becomes searchable for security questions: show all activity for user X in the last hour becomes a trace filter on the identity attribute.

What still needs a dedicated audit sink

Phoenix is operational observability. A few concerns require a separate path:

  • Long retention for compliance. Phoenix is typically tuned for days-to-weeks. Compliance often requires years. Forward security-relevant span attributes to an immutable log store, keyed by trace ID.
  • Tamper-evident storage. Traces are mutable infrastructure. High-value actions warrant append-only, signed audit records in a hardened store.
  • Stricter access control. Engineers debugging agent behavior should not incidentally see audit-grade identity data. The audit sink lives behind separate IAM.
  • Consent record references. If a user granted the agent specific permissions (especially for long-running workflows), the audit log should reference the consent grant ID alongside the action.

The simplified mental model

ConcernWhere it's solved
Operational + behavioral observabilityPhoenix / OTel — free with the stack
Security observabilityIdentity attributes on existing spans — cheap addition
Compliance / forensicsDedicated audit sink, fed from a subset of span attributes

The design decisions are about which attributes to attach where, and which to forward — not how to build an audit pipeline from scratch.

§ 12

Patterns to Establish

These are the principles worth codifying as bank-wide standards, not narrowly as AI patterns. They apply to every new workload deployed to the cluster, AI or otherwise.

  1. No long-lived secrets in pods. Workload identity via projected ServiceAccount tokens, federated to the IdP, is the only sanctioned pattern for new in-cluster workloads. Secret stores remain available for legacy and external integrations.
  2. Every service is an OAuth 2.0 resource server. Spring Security oauth2ResourceServer() with JWT validation, issuer pinned, audience enforced, scopes required for sensitive endpoints. No exceptions.
  3. User context flows via token exchange, never token forwarding. Including for non-AI services. The act claim in every downstream call produces a full delegation chain in audit records.
  4. Scopes describe capabilities, not roles. Resource-capability form (account:read). Roles stay in the IdP and map to scopes at token issuance.
  5. One exchange-client registration per downstream audience. Cheap to declare, expensive to skip. Audience-bound, scope-narrowed tokens are the default.
  6. Identity context on every trace span. user.id (or enduser.id, pick one and stick to it) and the act chain at minimum. Trace ID is the join key between operational observability and compliance audit.
  7. Token lifetimes calibrated to workload class. Workload tokens ~1 hour. User access tokens 15 minutes. Exchanged tokens inherit the shorter ceiling. Sensitive scopes require fresh acr.
  8. High-value actions require re-authentication. Step-up via acr claims, enforced at the downstream API. Agents cannot autonomously cross the threshold.

The architecture for AI workloads is the architecture for all modern workloads. The AI use case forces the conversation; the patterns benefit everything that comes after.

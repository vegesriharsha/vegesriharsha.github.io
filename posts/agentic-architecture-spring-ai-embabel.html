Agentic Architecture with Spring AI and Embabel

A working architect's notes on building agents on the JVM — composition over frameworks, hand-wired workflows in Spring AI, and goal-directed planning in Embabel — and the threshold that separates the two.

Audience
Platform, AI, and Application engineering teams
Scope
Conceptual foundations + implementation guidance
Status
Living document — patterns evolving
Framework versions assumed

Patterns and code in this document are anchored to: Spring Boot 3.5.x (Embabel's current ceiling) through 4.0 (Spring AI side), Spring AI 1.1.x (GA, Nov 2025) with 2.0 M-series in flight. The five workflow patterns and recursive advisors are core; higher-order patterns (Agent Skills, Todo, Subagent, AskUserQuestion) ship in the community spring-ai-agent-utils toolkit (group org.springaicommunity, ~0.4.x, requires Spring AI 2.0.0-M2+). Embabel is at 0.3.x-SNAPSHOT — past the 0.1.x stage noted in earlier evaluations, with GOAP + A* planning, a state-machine planner, Utility AI, and AgenticTool now present; module statuses range stable / incubating / experimental. Pin versions deliberately. This document deliberately does not re-cover memory or identity — see the companion documents for those planes.

§ 01

Context and Foundations

The companion documents in this series treat two of the planes an agent runs on: memory (what it remembers) and identity (who it acts as). This document treats the third and most contested plane: control flow — how an agent decides what to do next. That is the plane where the framework choice actually bites, and where the gap between “a chatbot with tools” and “an agent” is decided.

For a decade the JVM answer to “orchestrate these steps” was a state machine, a workflow engine, or hand-written service code. Those answers do not disappear with AI — they are still correct for most of the work. What changes is the introduction of a component, the LLM, whose output is non-deterministic and whose job is sometimes to decide the control flow itself. The architectural question of this document is: when do you let the model decide, when do you decide for it, and what sits in between?

The honest framing, and the one this document commits to: most production “agents” are workflows. They are a known sequence of LLM calls and tool invocations, wired together in ordinary code, with the model supplying judgment at specific points rather than steering the whole flow. Anthropic's own “Building Effective Agents” guidance — which Spring AI's pattern library is built directly on — draws exactly this line between workflows (predefined paths) and agents (the model directs its own process). Reach for genuine autonomy only when the problem actually demands it.

An agent is not a thing you import. It is a control-flow decision: how much of “what happens next” you delegate to a non-deterministic component, and how much you keep in code you can read, test, and audit.

Two frameworks anchor the JVM conversation. Spring AI gives you the primitives — model access, tool calling, advisors, structured output — and leaves orchestration to you, which for hand-wired workflows is exactly right. Embabel, from Spring's own creator, adds a planning layer above those primitives: you declare actions and goals, and a non-LLM algorithm assembles the path. This document develops both, and the threshold between them.

§ 02

What an Agent Actually Is

Stripped of marketing, an agentic system has four moving parts. Naming them precisely prevents the most common design error — treating “agent” as an atomic noun rather than a composition of separable concerns.

PART 01
The Model Call
A single inference: a prompt in, a completion out. Stateless, non-deterministic. Spring AI: ChatModel / ChatClient.
PART 02
Tools
The bridge to the outside world — APIs, databases, MCP servers. The model requests; code executes and returns. Spring AI: @Tool, ToolCallback, ToolCallingManager.
PART 03
Control Flow
What runs next, and why. The contested plane. Either you decide (workflow) or the model decides (autonomous agent) — or a planner decides (Embabel).
PART 04
Context Assembly
What goes into each call — memory, retrieved knowledge, prior results. Treated fully in the Memory document; here it is an input to control flow.

The first two parts are settled engineering. ChatClient wraps the model behind a fluent, WebClient-style builder; @Tool-annotated methods expose Java behavior to the model and participate in Spring Security and Micrometer like any other bean. The interesting decisions all live in part three — and that is where the rest of this document concentrates.

The definition that matters operationally

A useful working line: it is a workflow if you can draw the control-flow graph before the program runs, and an agent if you cannot — because the model (or a planner) determines the graph at runtime. This is not a value judgment. Workflows are easier to test, cheaper to run, and trivial to audit. The graph being knowable in advance is a feature, not a limitation, in a SOX-regulated environment. Choose autonomy only where the unpredictability of the problem genuinely requires an unpredictable solution.

§ 03

The Autonomy Spectrum

It helps to place every candidate design on a single axis, from least to most delegated control. The cost, testability, and auditability of a system degrade monotonically as you move right. The correct design is the leftmost point that solves the problem.

LevelWho decides next stepSpring AI / Embabel constructAuditability
0 — Single callYou (one step only)ChatClient.call()Trivial
1 — Fixed chainYou (static sequence)Chain workflowTrivial
2 — ConditionalYou + model classifiesRouting workflowHigh
3 — Bounded loopModel, within a coded loopEvaluator-Optimizer; recursive advisorHigh (loop is bounded)
4 — Delegated decompositionModel decomposes; you run workersOrchestrator-WorkersModerate
5 — PlannedNon-LLM planner over declared actionsEmbabel GOAPHigh (plan is explicit)
6 — Open autonomyModel, unconstrained tool loopRaw tool-calling agentLow

Two observations shape the whole document. First, auditability is not monotonic with autonomy — level 5 (Embabel) scores higher than level 4 precisely because the planner externalizes the decision into an inspectable plan rather than burying it in an LLM's reasoning. That non-monotonicity is the entire architectural argument for a planner in a regulated shop. Second, the levels are not a ladder you must climb; they are a menu. A realistic platform mixes them: a routing workflow (level 2) that dispatches to a fixed chain (level 1) for one branch and an Embabel plan (level 5) for another.

The level-6 trap

The unconstrained “give the model all the tools and let it loop until done” pattern is the easiest to demo and the hardest to operate. It has unbounded cost, unbounded latency, no pre-runtime control-flow graph, and an audit story that amounts to “read the model's chain of thought.” It is occasionally the right answer for genuinely open-ended exploration. It is almost never the right answer for a bounded business process, which is what nearly all enterprise work is.

§ 04

Spring AI Building Blocks

Before the patterns, the primitives they compose from. Spring AI deliberately keeps these small and orthogonal — the framework's bet is that orchestration belongs in your code, not behind a magic facade. Four building blocks carry almost everything.

ChatClient — the fluent entry point

ChatClient is the high-level API: a builder-style fluent interface over a ChatModel, modeled on WebClient. It is where you attach a system prompt, user message, advisors, tools, and a structured-output target. For programmatic and batch contexts, you can drop to the ChatModel directly — the same pattern the Memory document uses for non-ChatClient agent loops.

String answer = chatClient.prompt()
    .system("You are a claims-triage assistant.")
    .user(userMessage)
    .call()
    .content();

Advisors — middleware for the AI call

Advisors are the interceptor chain for a model interaction — the AI-layer analogue of servlet filters or Spring AOP. Each advisor can mutate the request before the call and the response after it. This is how RAG (QuestionAnswerAdvisor), conversation memory (MessageChatMemoryAdvisor), logging, and guardrails attach without touching business logic. The Advisors API integrates with Micrometer, so every advised call is observable for free.

Why advisors are an architectural primitive, not a convenience

The advisor chain is the seam where cross-cutting concerns live. Memory composition, retrieval, identity-aware filtering, audit emission, and output validation all attach as advisors rather than being interleaved into orchestration code. A discipline worth adopting: if a concern applies to every call, it is an advisor; if it is the call's actual purpose, it is orchestration. Keeping that line clean is what stops a workflow from collapsing into an unmaintainable tangle.

Tool calling — the action primitive

A @Tool-annotated method is a typed, described capability the model may request. Spring AI handles the JSON-schema generation, the call dispatch, and (by default) the execution-and-resubmit loop. Crucially for programmatic control, that loop is defeatable: setting internalToolExecutionEnabled(false) on ToolCallingChatOptions hands you the raw tool-call requests so you can execute them yourself via ToolCallingManager — the basis for any custom or audited agent loop.

// Hand-controlled tool loop — no ChatClient sugar, full audit control
var options = ToolCallingChatOptions.builder()
    .toolCallbacks(readOnlyTools)
    .internalToolExecutionEnabled(false)   // we drive the loop
    .build();

Prompt prompt = new Prompt(messages, options);
ChatResponse response = chatModel.call(prompt);

while (response.hasToolCalls()) {
    ToolExecutionResult result =
        toolCallingManager.executeToolCalls(prompt, response);
    auditEmitter.record(result.toolCallResults());   // audit every call
    prompt = new Prompt(result.conversationHistory(), options);
    response = chatModel.call(prompt);
}

Structured output — the typing primitive

BeanOutputConverter and the .entity(Class) shortcut turn a free-text completion into a typed Java record, with the schema injected into the prompt and the response parsed and validated on return. This is the single most important reliability lever in the toolkit: every boundary between an LLM call and your code should be a typed object, never a raw string. Typed boundaries are what make a multi-step flow testable — you assert on records, not on prose.

Strings are where agentic systems rot. Type every boundary. A record between two steps is a contract you can test; a string between two steps is a prayer.

§ 05

The Five Workflow Patterns

The five patterns below — codified by Christian Tzolov for Spring AI directly from Anthropic's “Building Effective Agents” — are the workhorses. They are not framework features; they are shapes of ordinary Java code that orchestrate LLM calls. You implement them with plain control flow, CompletableFutures, and typed records. This is the level where the vast majority of FHLB's bounded, deterministic use cases live.

The Five Patterns by Control Shape
1
Chain
Sequential steps, each consuming the prior output. Decompose a hard task into ordered, individually-simpler LLM calls. Use a programmatic gate between steps to fail fast on bad intermediate output. Best when the sub-steps are known and ordered.
2
Parallelization
Independent calls run concurrently (thread pool / CompletableFuture), then aggregate. Two sub-shapes: sectioning (split work into independent pieces) and voting (run the same prompt N times and take a majority). Best for throughput or diverse perspectives.
3
Routing
A classifier call inspects the input and dispatches to one of several specialized handlers/prompts. Separates classification from handling so each prompt stays focused. Best when inputs fall into distinct categories needing different treatment.
4
Orchestrator-Workers
A central LLM decomposes a task into subtasks at runtime, delegates each to a worker call, and synthesizes the results. The subtasks are not known in advance — that is the distinguishing feature versus Chain. First step onto genuine delegation (level 4).
5
Evaluator-Optimizer
A generator produces a draft; an evaluator scores it and returns feedback; the loop repeats until the evaluator passes or a bound is hit. A dual-LLM refinement loop — the most common bounded autonomy pattern. Best when quality criteria are expressible but hard to hit in one shot.

The decision among them

The choice is mechanical once framed correctly:

  • Are the steps known and ordered? → Chain.
  • Are the steps known and independent? → Parallelization.
  • Does the input fall into categories? → Routing.
  • Are the steps unknown until you see the input? → Orchestrator-Workers.
  • Is it one output that needs iterative refinement against a quality bar? → Evaluator-Optimizer.

These compose freely. A real email-triage flow might route by message type, run a chain for the routine branch, and invoke an evaluator-optimizer loop for the branch that drafts a customer response. Each pattern is a few dozen lines of testable Java; the composition is your application's actual logic.

Where these live relative to Kestra

These patterns govern control flow within a single agent invocation — the synchronous reasoning a service does to answer one request. They are not a replacement for Kestra, which owns durable, cross-process orchestration: retries across hours, human-in-the-loop waits, fan-out across many agents, and the audit trail of a long-running business process. The division of labor: Spring AI patterns own the in-process reasoning loop; Kestra owns the durable workflow that may invoke many such loops. Conflating the two — putting a multi-hour durable wait inside an in-process evaluator loop — is a recurring mistake.

§ 06

Recursive Advisors and the Agentic Loop

Spring AI 1.1 added recursive advisors — advisors that can re-enter the advisor chain, enabling controlled iteration without hand-writing the loop in business code. This is the framework's sanctioned mechanism for bounded autonomy: the iteration lives in a reusable, observable advisor rather than scattered through orchestration.

The pattern matters because it cleanly expresses three things that previously required custom loops:

  • Output validation with retry — generate, validate, and if invalid, re-prompt with the validation feedback, up to a bound. The same shape as Evaluator-Optimizer, but packaged as middleware.
  • LLM-as-a-Judge — a second model scores the first model's output against a rubric; the score gates acceptance or triggers another iteration. The basis for automated quality evaluation in production.
  • Self-organizing memory and guardrails — the recursion point is where a guardrail can reject and force regeneration, or where context can be re-summarized mid-flow.
Recursion needs a hard bound

Any recursive or iterative agentic construct must carry an explicit maximum-iteration cap and ideally a token/cost budget. A recursive advisor whose exit condition depends solely on an LLM judgment (“is this good enough yet?”) can loop indefinitely on adversarial or ambiguous input. The cap is not a fallback — it is a first-class part of the design. Emit an audit event when the cap is hit rather than the quality bar; a flow that repeatedly hits its cap is telling you the prompt or the criteria are wrong.

Recursive advisors are the natural ceiling of what Spring AI offers before you reach for a dedicated planning framework. They give you bounded, observable loops with retry and evaluation — which covers a large fraction of “agentic” requirements. What they do not give you is planning: the system still follows a path you wrote, just with iteration. The moment the requirement becomes “assemble a novel sequence of steps based on what's available,” you have crossed into Embabel's territory (§09).

§ 07

Higher-Order Patterns: Skills, Todo, Subagents

Beyond the core, the Spring AI community ships spring-ai-agent-utils (group org.springaicommunity) — a toolkit of higher-order agentic patterns inspired by Claude Code's design. These are pre-1.0 and require Spring AI 2.0.0-M2+, so treat them as forward-looking rather than load-bearing today, but they signal where the platform is heading and several map directly onto FHLB's roadmap concerns.

Their value now is conceptual: they name the patterns (skill loading, explicit todo planning, context-isolated subagents, dynamic tool search) that the platform will eventually standardize, which lets you build today's hand-wired equivalents in a shape that will migrate cleanly when the toolkit stabilizes.

§ 08

Where Hand-Wired Orchestration Strains

The workflow patterns are excellent until a specific kind of complexity appears. Recognizing that inflection is the whole point of evaluating Embabel — adopting a planner before the inflection is over-engineering; adopting it after is a painful rewrite. Four signals, in order of increasing significance:

Signal 1 — Combinatorial branching

A routing workflow with three branches is clean. A routing workflow that routes, then each branch conditionally routes again, then those sub-branches recombine, is a nested if/switch structure that no longer fits on a screen and resists testing. When your hand-wired control flow starts to look like a state machine you are maintaining by hand, you are paying the cost of planning without its benefits.

Signal 2 — The “recombine known steps in a new order” requirement

This is the decisive one. If a new requirement is satisfiable by running existing steps in a sequence you did not previously code — and you find yourself adding yet another orchestration method to wire that specific path — you are hand-doing what a planner does automatically. A planner derives the path from the steps' input/output types; you should not be writing the path at all.

Signal 3 — Auditability of the decision, not just the outcome

When compliance asks not “what did the agent do” but “why did it choose to do that, and what else could it have done,” an LLM-driven orchestrator's answer (“the model decided”) is unsatisfying. A planner produces an explicit plan object — the considered actions, their preconditions, the chosen sequence — which is a far stronger audit artifact.

Signal 4 — Adaptive replanning mid-flow

When a step's result should change which steps run next — not via a coded branch you anticipated, but because the new information unlocks or forecloses actions — you need replanning. Hand-wiring every “if this result, then reconsider everything” case is intractable past a few steps.

The threshold is not scale and not LLM count. It is the moment the path itself becomes something you would rather derive than write. Below it, Spring AI's patterns are the lighter, cheaper, more testable choice. Above it, you are hand-maintaining a planner.

Do not cross the threshold prematurely

The failure mode in the other direction is real: adopting a planning framework for a flow that is genuinely a fixed three-step chain adds a dependency, a programming model, a maturity risk (Embabel is 0.3.x), and a cognitive tax on the team — to derive a path that was never in doubt. A linear “classify → extract → route” flow does not need GOAP. Most flows do not. The planner earns its place only when several of the signals above are present together.

§ 09

The Embabel Model: Planning over Wiring

Embabel — created by Rod Johnson, Spring's founder, and built on top of Spring AI — inverts the orchestration question. Instead of writing the control-flow graph, you declare a set of actions (typed steps) and one or more goals (desired end states), and the framework's planner assembles a path from current state to goal. You describe the pieces; the system figures out the sequence.

The programming model is deliberately Spring-shaped. Agents are Spring-managed beans annotated @Agent; steps are methods annotated @Action; the terminal step that completes a goal is marked @AchievesGoal. There is also an idiomatic Kotlin DSL (agent { action { ... } }) for programmatic composition, but the annotation model is the one a Spring team reads fluently on day one.

@Agent(description = "Triage an inbound member email to a resolution")
public class EmailTriageAgent {

    @Action
    public Classification classify(InboundEmail email, OperationContext ctx) {
        return ctx.ai().withDefaultLlm()
            .createObject("Classify this email: " + email.body(),
                          Classification.class);
    }

    @Action
    public EnrichedCase enrich(Classification c, MemberRecord member) {
        // ordinary code + tool calls; no LLM required for this step
        return CaseEnricher.from(c, member);
    }

    @AchievesGoal(description = "Produce a routed, enriched triage result")
    @Action
    public TriageResult route(EnrichedCase enriched, OperationContext ctx) {
        return ctx.ai().withDefaultLlm()
            .createObject("Decide routing for: " + enriched, TriageResult.class);
    }
}

The critical thing to notice: there is no orchestration code. No method calls classify then enrich then route. Embabel reads the type signatures — classify produces a Classification, enrich consumes one and produces an EnrichedCase, route consumes that and achieves the goal — and derives the sequence. Add a new action that produces an EnrichedCase by a different route, and the planner can use it without any change to the existing flow.

The differentiator, precisely stated

Most frameworks wire control flow one of two ways: let the LLM pick the next tool at runtime (sacrificing predictability), or define a state machine ahead of time (LangGraph's approach). Embabel does neither. It uses GOAP — Goal-Oriented Action Planning, a non-LLM pathfinding algorithm from game AI — to select the execution path over your strongly-typed methods. The decision of what runs next is made by a deterministic, inspectable algorithm, not by the model and not by a graph you hand-drew. The model is still used inside actions; it just no longer steers the flow.

This is also where the type system does real work. Because actions are connected by their Java/Kotlin types — records, not string keys — the plan is checked against compile-time guarantees that data flows correctly between steps. Johnson's framing is that Java's type system is an asset for agentic AI that Python cannot match: prompts and return types are strongly typed, tool-able, and survive refactoring.

§ 10

GOAP, the Blackboard, and the OODA Loop

Understanding Embabel operationally means understanding three mechanisms: how it plans, where it keeps state, and how it loops.

GOAP planning with A*

After each action executes, Embabel replans. The cycle: examine current state, identify which actions' preconditions are now satisfiable, run an A* search over possible action sequences to find an optimal path to the goal, select the best plan by cost and success probability, execute the first action, then replan. Planning is not a one-time compile step — it happens continuously, which is what lets the system adapt when an action's result changes what is possible.

The Embabel Execution Cycle (OODA)
1
Observe
Read the current blackboard state and the results of the last action. What data objects now exist?
2
Orient
Determine what has changed since the last cycle and which actions' preconditions are now met.
3
Decide
Run A* over available actions to (re)formulate the optimal plan toward the goal. The plan is an explicit object.
4
Act
Execute the first action of the plan. Its output lands on the blackboard. Loop back to Observe.

The blackboard

State lives in a blackboard — a shared workspace that accumulates the typed data objects produced by actions over the run. An action's preconditions are essentially “these types are present on the blackboard”; its effect is “this type is now present.” This is what makes planning tractable and inspectable: the planner reasons over the presence and absence of typed objects, and the blackboard is the audit record of what the agent knew at each step.

Why replanning is a safety feature

The replan-after-every-action loop is not just flexibility — it is resilience. If an action fails or returns something unexpected, the planner reassesses from the actual current state and may find an alternative path to the goal that the original plan did not anticipate. A hand-wired chain would simply throw; the planner routes around. For a compliance-sensitive flow, the combination of “explicit plan object” plus “blackboard state at each step” produces an audit trail that answers not just what happened but what the system considered and why it chose its path.

If an agent gets stuck — no action's preconditions are satisfiable and the goal is unreached — an agent class implementing the StuckHandler interface can intervene, for example by adding data to the blackboard to unblock planning. This is the sanctioned escape hatch for partial or degraded inputs.

§ 11

Planner Types and Composition

A common early objection — “GOAP is elegant but my flow has a loop, and loops are awkward to express purely in preconditions” — has been substantially answered by Embabel's recent work. The framework now supports multiple planner types that compose, which materially changes the adoption calculus from the 0.1.x era.

PlannerSelects actions byBest for
GOAP (default)A* search over preconditions/effects toward a goalGoal-directed flows where the path should be derived from available actions
State machineExplicit states and transitionsLoops and stage-based flows that are naturally stateful (chatbots, multi-turn)
Utility AIDynamic utility scores per action — maximize overall utilityOpen-ended exploration where there is no single goal to reach, just value to maximize

The significant part is that these are not mutually exclusive. The programming model is identical across planner types, and they interoperate: an action run under the GOAP planner that returns a state type can hand control to the state machine, whose ultimate return passes back to GOAP — and each state defaults to GOAP planning internally. This is Embabel's claim to implement LangGraph's core idea (explicit stateful graphs) while keeping the planning advantages, and it removes the “loops are hard in GOAP” objection by letting the state machine own the loop and GOAP own the goal-seeking.

AgenticTool — the familiar supervisor pattern

For teams arriving from LangChain or Google ADK, Embabel ships AgenticTool: an LLM orchestrating sub-tools, mapping directly onto LangChain's supervisor-with-workers or ADK's coordinator pattern. This matters for adoption — it means Embabel is not an all-or-nothing bet on GOAP. You can use the familiar supervisor shape where it fits and the planner where it adds value, in the same application.

Module maturity is not uniform

Embabel labels its modules stable (production-ready, breaking changes avoided), incubating (usable, may break in minor releases), or experimental (early, may break in any release). When evaluating, check the status of the specific modules a design depends on rather than treating “Embabel 0.3.x” as a single maturity level. The core annotation model and GOAP planner are the most settled; newer planner integrations and tooling are further left on the maturity scale.

§ 12

How Spring AI and Embabel Relate

The single most important fact for avoiding a false choice: Embabel is built on Spring AI, not instead of it. It is not a competitor to evaluate against Spring AI; it is a higher-level layer that consumes Spring AI's model abstraction underneath. The relationship is layered, and both layers are present in an Embabel application.

The Layering
4
Embabel — Planning
@Agent / @Action / @AchievesGoal, GOAP/state-machine/utility planners, blackboard, replanning. Decides what runs next.
3
Spring AI — Orchestration primitives
ChatClient, advisors, tool calling, structured output. Embabel Llm beans wrap Spring AI ChatModel instances directly.
2
Spring AI — Model abstraction
Provider-agnostic ChatModel / EmbeddingModel. Any provider with a Spring AI ChatModel is usable from Embabel.
1
Transport
LiteLLM proxy → NIM models on OpenShift. Unchanged whether you use Spring AI alone or Embabel on top.

Johnson's own analogy is the cleanest: Spring AI sits at the level of the Servlet API; Embabel is more like Spring MVC. You can write a web application directly against servlets, and for a trivial endpoint that is fine. For anything with real structure, the higher-level framework expresses the intent more directly and tests more easily. The same relationship holds here. A fixed chain is a servlet; a goal-directed, replanning, multi-branch flow is what Spring MVC is for.

Practical consequence for the platform

Because Embabel consumes Spring AI's ChatModel, the existing transport investment is preserved entirely: an Embabel Llm bean is created around a Spring AI ChatModel, so the LiteLLM-proxy-to-NIM path, model selection, and observability all carry over. Adopting Embabel for a specific agent does not fork the platform's model-access story. It adds a planning layer to that agent while leaving every other agent on plain Spring AI. The two coexist service-by-service.

This also dissolves the “Spring AI vs Embabel” framing into the right question: per agent, does this flow need a planner? Most will not, and run on Spring AI's patterns. A minority — the compliance-sensitive, multi-branch, replanning flows — will, and run on Embabel over the same Spring AI foundation.

§ 13

Agent Topology and Deployment

The framework choice interacts with the deployment topology. The platform's standing decision — one Spring Boot microservice per agent, breaking even around agent ten because the per-agent ceremony is templated platform investment rather than manual work — holds regardless of whether a given agent uses plain Spring AI or Embabel. But the control-flow choice changes what lives inside each service.

What a Spring-AI-only agent service contains

A ChatClient (or raw ChatModel loop), an advisor chain, a set of @Tool beans, and orchestration code expressing one or more of the five patterns. The control-flow graph is in your source, reviewable in a PR. Startup is fast; the dependency surface is small. This is the default and the bulk of the fleet.

What an Embabel agent service adds

The Embabel starter, @Agent-annotated beans whose @Action methods are scanned at startup into runtime metadata, and the planner. The control-flow graph is not in your source — it is derived at runtime from action signatures. This is the trade: you gain derived, adaptive paths and lose the ability to read the exact sequence from the code. The Spring-Boot-3.5.x constraint on Embabel currently means an Embabel service may lag the Spring Boot 4.x line that plain Spring AI services can adopt — a real version-skew consideration for a mixed fleet.

Version skew across the fleet

Mixing Spring AI services on Spring Boot 4.x with Embabel services pinned to 3.5.x is workable but must be deliberate. Shared libraries (security, observability, identity propagation per the Identity document) need to support both lines, or the Embabel services need their own compatible versions. Track Embabel's Spring Boot 4 / Spring AI 2.0 support actively; the constraint is a moving target and may lift before a broad Embabel rollout. Until it does, treat Embabel adoption as scoped to specific high-value agents rather than a fleet-wide default.

Identity and memory propagation are unchanged

Both layers run inside an ordinary Spring Boot service, so the OAuth2 on-behalf-of token propagation from the Identity document and the four-plane memory composition from the Memory document apply identically. An Embabel @Action that calls a tool propagates the caller's Authentication exactly as a Spring AI tool call does — the planner does not bypass the security or memory machinery, it sits above it. This is a direct benefit of Embabel embracing the Spring component model rather than replacing it: dependency injection, AOP, transaction management, and security all work as in any Spring application.

§ 14

Determinism, Auditability, and Cost

Three operational properties separate the approaches in ways that matter directly to a SOX-regulated platform. None is uniformly better; the right choice depends on which property a given flow most needs.

Determinism

No approach makes the model deterministic — that is inherent to LLMs. What varies is whether the control flow is deterministic. Fixed chains and routing are fully deterministic in structure. Embabel's planning is deterministic-given-state: the A* search over the same blackboard state and action set yields the same plan, so the path is reproducible and explainable even though the model outputs inside actions are not. An open tool-calling loop is non-deterministic in both structure and content — the worst case for reproducibility.

Auditability

This is where the planner's advantage is sharpest. A hand-wired workflow audits well at the outcome level (you logged each step) but poorly at the decision level for branches the model drove. Embabel produces an explicit plan object plus a blackboard history, so the audit answers “what did the system consider, what were the preconditions, why this path” — not just “what ran.” For the examination-tracking and email-triage flows where an auditor may challenge a routing decision, this is a material difference.

The audit artifact comparison

For a SOX review, the difference is concrete. A Spring AI orchestrator's audit trail is a sequence of structured log events: step ran, input, output. An Embabel run additionally yields the plan that was formulated (the considered action sequence with preconditions) and the blackboard at each step (the typed state the decision was made against). The former proves what happened; the latter additionally documents the reasoning structure — closer to what an auditor actually wants to see for an automated decision.

Cost and latency

More structure can mean more model calls. Evaluator-Optimizer loops, orchestrator decomposition, and replanning all add calls relative to a single shot. The lever Embabel offers here is real: because flows decompose into small, focused actions, each action can use a smaller, cheaper model sized to its specific sub-task, rather than one large model for the whole flow. Embabel encourages per-invocation model choice via its PromptRunner / role mechanism. The net cost can be lower than a monolithic prompt even with more calls — but only if you exploit the per-action model sizing rather than defaulting every action to the largest model.

Auditability is not the price of autonomy — at the planner level it is a product of it. The explicit plan is both the thing that makes the agent adaptive and the thing that makes it explainable. That coincidence is the strongest single argument for planning in a regulated environment.

§ 15

A Decision Framework

Pulling the document into a single procedure. For any candidate agent, walk the questions in order and stop at the first that fits — the leftmost point on the autonomy spectrum that solves the problem.

Choosing the Control-Flow Approach
1
One call?
If the task is a single prompt-and-response (classify, extract, summarize), use ChatClient directly. No orchestration. Stop.
2
Known fixed path?
If the steps and their order are known at design time, use the Chain / Parallelization / Routing patterns in plain Spring AI. This covers most bounded business processes. Stop.
3
Iterative refinement?
If it is one output that must meet a quality bar, use Evaluator-Optimizer or a recursive advisor with a hard iteration cap. Still Spring AI. Stop.
4
Runtime decomposition?
If subtasks are unknown until input arrives but the flow is otherwise simple, use Orchestrator-Workers. Reassess if the orchestration logic starts to sprawl.
5
Planning signals present?
If two or more §08 signals hold — combinatorial branching, recombine-known-steps, decision-level audit, adaptive replanning — evaluate Embabel for this agent. Otherwise stay on Spring AI.
6
Durable / cross-process?
If the flow spans hours, needs retries across failures, or human waits, that orchestration belongs in Kestra regardless of the in-process choice above.

The framework is intentionally biased toward the left. The reason is the asymmetry of regret: starting on Spring AI patterns and later wrapping a planner around a flow is an additive change; starting on a planner for a flow that never needed one is a dependency and a maturity risk you carry for the life of the service. When genuinely uncertain, build the Spring AI version first — it is also the fastest way to learn whether the planning signals are real or imagined.

§ 16

Failure Modes

The characteristic ways agentic control flow goes wrong. Most are not exotic — they are ordinary engineering failures wearing an AI costume.

The unbounded loop

Any loop whose exit depends on a model judgment can run forever on adversarial input. Every iterative construct — evaluator-optimizer, recursive advisor, planner replanning — needs a hard iteration cap and a cost budget, with an audit event emitted when the cap (rather than the success condition) is what stops it.

The string-typed boundary

Passing raw model text between steps instead of parsed, typed records. The flow appears to work in the demo and fails silently in production when the model phrases something differently. Every inter-step boundary should be a record produced by structured output, asserted on in tests.

Premature autonomy

Reaching for an orchestrator or a planner when a fixed chain would do. Symptoms: a control-flow design discussion that is longer than the flow would be to hand-write; a planner modeling a path that was never in doubt. The cost is paid in latency, dollars, and debuggability.

Premature framework lock-in

The inverse: hand-wiring a state machine in nested conditionals because “we don't need a framework,” until the conditionals are unmaintainable. The §08 signals exist to catch this before the rewrite is forced.

Planner mismodeling

An Embabel-specific failure: declaring action preconditions/effects (via types) such that no valid plan exists, or such that the planner picks a surprising path. Manifests as “the agent won't start” or “the agent does something reasonable but not what I expected.” The fix is the same discipline as good type design — and a StuckHandler for the partial-input cases.

Confusing in-process and durable orchestration

Putting a multi-hour wait or a human-approval step inside a synchronous in-process reasoning loop, where it blocks a thread and loses its state on restart. Durable, cross-process orchestration is Kestra's job; the in-process patterns assume they run to completion within one request.

The meta-failure

Underlying most of the above is treating “agent” as a destination rather than a control-flow choice. Teams decide to “build an agent” and then reach for the most autonomous construct available, when the task was a three-step workflow all along. Start from the task's actual shape and pick the leftmost construct that fits — the autonomy is a cost to be justified, not a goal to be reached.

§ 17

Patterns to Establish

The standing decisions this document argues for, collected for reference. These are platform conventions, not per-project choices.

  1. Default to workflows, not autonomy. The leftmost point on the autonomy spectrum that solves the problem is the correct one. Genuine autonomy is opt-in and justified, never the default.
  2. Spring AI's five patterns are the workhorse layer. Chain, Parallelization, Routing, Orchestrator-Workers, Evaluator-Optimizer — implemented as plain, testable Java orchestrating LLM calls. Most agents never need more.
  3. Type every boundary. Structured output between every step. Records, not strings. A boundary you cannot assert on in a test is a defect.
  4. Every loop is bounded. Hard iteration cap and cost budget on every iterative or recursive construct. Emit an audit event when the cap stops the loop.
  5. Cross-cutting concerns are advisors; the task is orchestration. Memory, retrieval, identity filtering, audit, guardrails attach as advisors. Keep that line clean.
  6. Embabel is a per-agent decision, evaluated against the §08 signals. Not a fleet default and not a competitor to Spring AI — a planning layer adopted for specific flows that show two or more planning signals together.
  7. Embabel runs on the existing Spring AI foundation. Its Llm beans wrap Spring AI ChatModel instances; the LiteLLM/NIM transport, identity propagation, and memory composition carry over unchanged.
  8. In-process control flow is not durable orchestration. The patterns here own the synchronous reasoning loop. Anything spanning processes, hours, retries, or human waits belongs in Kestra.
  9. Per-action model sizing. Decompose so each step can use the smallest model that suffices. Do not default every call to the largest model; the decomposition exists partly to let you avoid that.
  10. Track maturity explicitly. Embabel is 0.3.x with mixed module statuses; spring-ai-agent-utils is pre-1.0 on Spring AI 2.0 milestones. Pin versions, check module status per dependency, and keep these scoped to where their value is clear.

The architecture for agentic control flow is the architecture for any system that mixes deterministic code with a non-deterministic component: isolate the non-determinism, type its boundaries, bound its loops, and keep the decision of “what runs next” as explicit as the problem allows. The AI use case forces the discipline; the discipline is just good engineering.

§ 18

POC Injection Points

Each POC below is a self-contained scope exercising one concept from this document. The Claude Code prompt blocks are starting points — paste them into a Claude Code session, refine with environment-specific detail (NIM endpoint and model identifiers via the LiteLLM proxy, tenant identifiers, downstream service names), and iterate. POCs are ordered from the workflow layer to the planning layer; later POCs assume the foundational ones are understood. They deliberately do not re-derive memory or identity scaffolding — wire those in per the companion documents where noted.

POC 01
Routing + Chain workflow with typed boundaries
Scope · Level 1–2 — Hand-wired workflow
Prereqs — Spring Boot 3.5+ / Spring AI 1.1.x. Chat model endpoint reachable (NIM via LiteLLM proxy).

Demonstrates the two foundational patterns composed: classify an input, then run a fixed chain for the matched branch, with every inter-step boundary a typed record asserted in tests.

Create a Spring Boot 3.5+ service using Spring AI 1.1.x that implements a routing-then-chain workflow for inbound member-email triage. Requirements:

1. POST /api/triage accepts an InboundEmail record (from, subject, body).
2. Routing step: a classifier ChatClient call returns a typed Classification record (category enum: BALANCE_INQUIRY | DISPUTE | GENERAL, confidence double) via .entity(Classification.class).
3. Chain step: for DISPUTE, run a three-call chain — extract dispute details → enrich with a stubbed account-lookup tool → draft a structured DisputeCase record. Each call's output is a typed record consumed by the next.
4. Insert a programmatic gate between chain steps: if extraction confidence < 0.6, short-circuit to a NEEDS_HUMAN result rather than continuing.
5. All inter-step types are Java records; no raw strings cross a boundary.
6. Emit a structured audit event per step (step name, input hash, output summary, latency, token usage).
7. Tests: (a) a classification test asserting category for fixtured emails using a mocked ChatModel returning canned completions; (b) a chain test asserting the DisputeCase shape; (c) a gate test asserting the low-confidence short-circuit.

Produce: build.gradle, application.yml, the record types, TriageController.java, RoutingService.java, DisputeChain.java, an AuditEmitter, and the three tests.

POC 02
Parallelization — sectioning and voting
Scope · Level 1 — Concurrent workflow
Prereqs — POC 01 understood.

Demonstrates both parallelization sub-shapes: splitting independent work across concurrent calls, and running one prompt N times for a majority vote.

Extend the POC 01 service with two parallelization endpoints. Requirements:

1. Sectioning: POST /api/analyze-document takes a long document and runs three independent ChatClient calls concurrently via CompletableFuture on a bounded executor — one for summary, one for risk flags, one for required-action extraction — each returning a typed record. Aggregate into a DocumentAnalysis record.
2. Voting: POST /api/classify-sensitive runs the same classification prompt five times concurrently, parses each to a typed enum, and returns the majority verdict plus the vote distribution. Tie-break to the most conservative category.
3. The executor is a managed Spring bean with a fixed pool size and a sensible rejection policy; do not use the common ForkJoinPool.
4. Both endpoints carry a per-request timeout; on partial completion, return the sections that succeeded plus an explicit list of the ones that timed out.
5. Tests: (a) sectioning assembles all three sections; (b) voting returns the majority and distribution for a fixtured set of mocked responses; (c) a timeout test asserting graceful partial return.

Produce: ParallelAnalysisService.java, VotingClassifier.java, executor configuration, the record types, and the three tests.

POC 03
Evaluator-Optimizer loop with a hard bound
Scope · Level 3 — Bounded autonomy
Prereqs — POC 01 understood.

Demonstrates the dual-LLM refinement loop and, critically, that it is bounded and audited — the loop stops on either the quality bar or the iteration cap, and reports which.

Build an evaluator-optimizer service that drafts a member-facing response and refines it against a rubric. Requirements:

1. Generator: a ChatClient call produces a Draft record from a case summary.
2. Evaluator: a separate ChatClient call (different system prompt, optionally a different/smaller model) scores the draft against an explicit rubric and returns an Evaluation record (passed boolean, score, feedback list).
3. Loop: generate → evaluate → if not passed, regenerate with the feedback injected, up to maxIterations=4 (configurable). The cap is mandatory.
4. The result records: final draft, number of iterations, and whether termination was by QUALITY_BAR or ITERATION_CAP. Emit a distinct audit event for cap-termination.
5. Add a cost budget: track cumulative token usage across iterations and terminate early if a configured ceiling is exceeded, recording BUDGET as the termination reason.
6. Implement it twice: once as plain orchestration code, and once as a Spring AI recursive advisor. Document the trade-offs in a README.
7. Tests: (a) terminates on quality bar when the mocked evaluator passes on iteration 2; (b) terminates on cap when the evaluator never passes; (c) terminates on budget when token usage exceeds the ceiling.

Produce: the record types, EvaluatorOptimizerService.java (orchestration version), QualityRefinementAdvisor.java (recursive-advisor version), configuration, README, and the three tests.

POC 04
Orchestrator-Workers with runtime decomposition
Scope · Level 4 — Delegated decomposition
Prereqs — POCs 01–02 understood.

Demonstrates the first genuinely delegated pattern: an orchestrator LLM that decomposes a task into subtasks not known at design time, dispatches them to workers, and synthesizes the result — while keeping the decomposition itself inspectable.

Build an orchestrator-workers service for ad-hoc document-package review. Requirements:

1. Orchestrator: a ChatClient call takes a review request and returns a typed Plan record — a list of Subtask records (type, instruction, targetSection) it has decided are needed. The set and number of subtasks are determined at runtime.
2. Workers: each subtask is dispatched to a worker ChatClient call (parallelized where independent) returning a typed SubtaskResult.
3. Synthesis: a final call combines the worker results into a ReviewReport record.
4. Inspectability: persist the orchestrator's Plan as a first-class audit artifact before any worker runs, so the decomposition decision is reviewable independently of the outcome.
5. Guardrails: cap the number of subtasks the orchestrator may create; reject and re-prompt once if it exceeds the cap; emit an audit event if it still does.
6. Tests: (a) a fixtured request produces the expected subtask set from a mocked orchestrator; (b) workers run and synthesis assembles the report; (c) the subtask cap is enforced.

Produce: the record types, OrchestratorService.java, WorkerService.java, the plan-persistence audit component, configuration, and the three tests.

POC 05
First Embabel agent — GOAP from action types

Prereqs — POCs 01–04 understood. Spring Boot 3.5.x (Embabel constraint). Embabel agent starter.

Demonstrates the core Embabel value: a multi-step flow with no orchestration code, where the planner derives the sequence from action input/output types and replans after each step.

Create a Spring Boot 3.5.x service using the Embabel agent starter that reimplements the POC 01 triage flow as a planned agent. Requirements:

1. Add the appropriate Embabel starter; configure an Embabel Llm bean wrapping the existing Spring AI ChatModel (NIM via LiteLLM) so the transport is shared with the rest of the platform.
2. Define an @Agent class with @Action methods: classify(InboundEmail) → Classification, enrich(Classification, MemberRecord) → EnrichedCase, and an @AchievesGoal action route(EnrichedCase) → TriageResult. 3. Add a second path to EnrichedCase (e.g., enrichFromCache(Classification) → EnrichedCase) and demonstrate that the planner can use it without any change to route — show both plans in the logs.
4. Implement StuckHandler on the agent to handle the case where no MemberRecord is available, adding a placeholder to the blackboard so planning can proceed to a degraded result.

Evaluating the Evaluators

Choosing an architecture for AI quality at the bank. RAGAs, DeepEval, and Promptfoo get pitched as alternatives, but really occupy three different layers of one stack — a starting recommendation, an end-state architecture, and a phased plan with paste-ready POCs.

Audience
AI, Platform, and Risk teams
Scope
Pre-deployment evaluation
Regulatory
SOX · FHFA examination
Status
Living document — draft for review
Framework versions assumed

Tooling and code in this document are anchored to: Promptfoo 0.121.11 (MIT; now part of OpenAI), Ragas 0.4.3 (Apache-2.0; floor ≥ 0.4.0 for a known CVE), DeepEval 4.0.0 (Apache-2.0; optional, phase 2), on Node.js 22.22 LTS and Python 3.12. Runtime observability continues on Phoenix (Arize), already deployed. Self-hosted CI runners on UBI 9 inside OpenShift. § 01

Executive Summary

Recommendation

Start with Promptfoo as the evaluation harness and Ragas as a Python-loaded metric provider inside it. Treat DeepEval as a deferred option for criteria-driven (G-Eval) metrics in phase two. Keep Phoenix where it is — it is the runtime observability layer, not a pre-deployment eval framework.

This is not a single-tool decision. It is a stack decision: a declarative harness on top, a swappable metric library in the middle, FHLB-specific compliance evaluators at the bottom, with Phoenix continuing to handle production traces in parallel.

The reasoning is straightforward for our environment: a SOX-regulated bank needs evals that are diff-able in a pull request, reproducible in CI, and signed off by reviewers. Promptfoo's YAML-first design gives us that. Ragas supplies the RAG-specific metrics where the Promptfoo built-ins are lighter. DeepEval's G-Eval is a powerful pattern for SME-authored criteria — but it is something we can add once the harness exists, not a prerequisite.

The question is not which tool. The question is which layer each one owns.

§ 02

The Question, Reframed

Before picking a tool, define what we mean by "evaluation." The word is overloaded, and the three frameworks live in different parts of it.

At least three distinct activities get bundled under one word at most banks. They have different lifecycles, owners, and tooling — and conflating them is what makes the tool choice feel harder than it is.

ACTIVITY 01
Pre-deployment
A prompt, model, retriever, or embedding changes. Before it ships, run a curated dataset through it and verify quality did not regress. Closed-world, reproducible, gating.
Promptfoo · Ragas · DeepEval
ACTIVITY 02
Runtime observability
The system is live. Traces, latency, token cost, retrieval hit-rate, drift. Open-world, statistical, observational. A different problem with a different tool.
Phoenix (already in place)
ACTIVITY 03
Adversarial / red-team
Generate inputs designed to break the system — injection, jailbreaks, PII extraction, policy circumvention. Overlaps pre-deployment, but with adversarial rather than golden datasets.
Promptfoo (50+) · DeepEval (40+)
Why this reframe matters

Treating "Promptfoo vs. RAGAs vs. DeepEval" as a three-way choice forces a single tool to do all three jobs, and at least one of them poorly. The right question is not which one, but which layer does each one own.

§ 03

Meet the Three Contenders

A short, honest read on each — what it actually is, where it shines, and where it is being asked to do something it was not designed for.

Promptfoo — the harness

An open-source LLM test harness: YAML-driven config, CLI-first, MIT-licensed, written in TypeScript with a Python wrapper. Config declares providers (model endpoints), prompts, test cases, and assertions; the CLI runs the matrix and emits JSON, HTML, and JUnit XML. Used internally by OpenAI and Anthropic. As of April 2026, Promptfoo became part of OpenAI; the project remains open source and MIT-licensed.

Strengths. Declarative config that diffs cleanly in PRs. First-class CI/CD support, especially GitHub Actions. Side-by-side prompt and model comparison UI. Strong red-team module with 50+ vulnerability plugins. Provider-agnostic — any OpenAI-compatible endpoint works out of the box, which matches our Triton deployment.

Honest weaknesses. Built-in assertions for RAG-specific metrics (faithfulness, context precision) are lighter than Ragas. The path to good RAG metrics is calling Ragas as a Python custom provider — fine, but worth knowing.

Ragas — the metric library

An open-source metrics library for RAG and, increasingly, agent evaluation. Apache-licensed, Python. What it does best: rigorously defined, research-backed implementations of RAG metrics — faithfulness, answer relevancy, context precision, context recall, noise sensitivity, entity recall.

Strengths. Best-in-class RAG metric implementations. Synthetic testset generation. Good integration with Spring-adjacent Python stacks, LangChain, LlamaIndex, Haystack, and standalone use.

Honest weaknesses. Not a harness — you write Python to drive it, and there is no "config + CI + report" story baked in. Treating Ragas as the primary tool means writing your own harness. Also: pin >= 0.4.0 — an arbitrary file read CVE existed in 0.2.3–0.2.14.

DeepEval — metrics with a pytest harness

An open-source "LLM unit-testing" framework, Apache-licensed, Python. Modeled on pytest — you write test_* functions asserting metric thresholds. 14+ built-in metrics across RAG, hallucination, bias, toxicity, summarization, plus a powerful pattern called G-Eval: the LLM judges output against criteria you write in natural language.

Strengths. G-Eval is genuinely useful — describe in English what "a good response" means, and the framework runs an LLM-as-judge against that rubric. Ideal for SME-authored, domain-specific checks. Good pytest ergonomics. Red-team module covers 40+ vulnerabilities. Active — v4.0 shipped May 2026.

Honest weaknesses. The harness layer (pytest) suits engineers, not non-engineer reviewers. Configs are Python, less reviewable than YAML for compliance audiences. Rich result hosting is a paid offering; self-hosted reporting is thinner.

§ 04

Why They Are Not Competitors

A harness orchestrates. A metric scores. A runtime observes. Three jobs — and the three frameworks each do one of them well.

The Pre-deployment Eval Stack

Fig. 1 — Pre-deployment eval as a four-layer stack, with Phoenix runtime observability as a parallel concern that closes the loop.

Once the layers are drawn explicitly, the integration story is obvious: Promptfoo invokes Ragas (and optionally DeepEval) as Python providers. The harness owns orchestration, CI, reports, comparison UI, and red-team flow. The metric libraries own the math. Custom Python evaluators handle the compliance assertions only we can write — PII rules tied to our data classification, NL2SQL guardrails specific to our warehouse, policy checks specific to FHLB advances.

§ 05

Mapping This to FHLB

The recommendation above is generic. Here is why it specifically fits us.

SOX & FHFA examination posture

Every eval configuration must be reviewable by a non-engineer — risk, audit, a compliance partner — and reproducible months later. YAML configs versioned in Git, signed off in pull requests, with retained build artifacts, satisfies this in a way Python-only frameworks make harder. Promptfoo's declarative output is the closest thing to a paper trail an examiner expects.

On-prem Nemotron via NIM/Triton

All three frameworks speak OpenAI-compatible endpoints. Our Nemotron-Nano-30B-A3B on the L40S, behind the FastAPI sidecar, exposes exactly that. Provider config is one stanza, and no data leaves the perimeter:

providers:
  - id: openai:chat:nemotron-nano-30b-a3b
    config:
      apiBaseUrl: https://triton.fhlb-internal/v1
      apiKey: ${TRITON_INTERNAL_KEY}

Spring AI orchestration

The harness lives outside the Spring application. Two patterns are available; we start with the first:

  • Endpoint-level eval (start here). Promptfoo calls the Spring AI service's REST endpoints as an HTTP provider. We evaluate the assembled system — retrieval plus prompt plus model — not just the model. This is what regulators actually care about: end-to-end behavior.
  • Component-level eval (later). Bypass Spring AI and hit Triton directly, swapping prompts and configs. Useful for isolating the LLM's contribution from the retriever's.

GitHub Actions migration

Our GitLab → GitHub move, already in flight, aligns cleanly. The promptfoo-action integration is first-class: PR comments with eval diffs, branch-protection gating, JUnit reports, artifact uploads.

pgvector & AGE

The warehouse and the vector store are the source of truth for datasets. Golden datasets are extracted from production queries, redacted, and committed to the eval repo. Synthetic datasets from Ragas testset generation land in a staging schema for SME review before promotion.

Phoenix is already in place

We do not replace Phoenix — it is the runtime side. Failures observed in Phoenix become candidate test cases for the pre-deployment suite; that is the only crossover. The two coexist without overlap.

Note — Promptfoo & the OpenAI acquisition

Promptfoo became part of OpenAI in April 2026. The project remains MIT-licensed and open source. Our usage is self-hosted; the CLI runs locally and sends nothing externally when PROMPTFOO_DISABLE_TELEMETRY=1 is set, which we will set globally in CI and in any developer wrappers. The acquisition does not, on its face, change our risk posture — but it warrants a periodic review of governance and license terms, and we should document a contingency in case licensing changes (the practical fallback is forking the last MIT release tag).

How this lands inside the bank's existing AI platform.

DeveloperCommits a prompt change, retriever tweak, or new model version; opens a PR against the eval repo.
2
GitHub ActionsTrigger fires. A self-hosted UBI 9 runner inside OpenShift picks up the job — network access to triton.fhlb-internal, no public egress (enforced by policy).
3

7

8

§ 07

What We Learned

The reasoning above, distilled into principles to carry through implementation.

1 — Eval is code

Configs, datasets, and assertions live in Git: diffed in PRs, reviewed by humans, signed off, versioned. The only practical way to make AI quality auditable in a SOX environment.

2 — Separate harness from metrics

The harness changes rarely; metrics evolve constantly as research advances. Locking them together — picking DeepEval and writing all assertions in its Python API — makes swapping metric libraries expensive later. A YAML harness with Python providers is loosely coupled by design.

3 — Pre-deployment is not runtime observability

Pre-deploy is closed-world, reproducible, gating. Runtime is open-world, statistical, observational. Phoenix and Promptfoo each do one well. Conflating them produces worse evals and worse observability.

4 — Start with regression, not red-team

The first golden dataset is worth more than the first adversarial probe. You cannot tell whether a red-team finding is a real regression without a baseline. Golden suite first; red-team second.

5 — SMEs author criteria; engineers wire the harness

The most valuable evals are written by people who know what a good answer looks like — Capital Markets analysts, Member Services examiners, the policy team. G-Eval (rubric-in-English, judged by an LLM) is the right shape for SME-authored evals. Engineering builds the rails; the business writes the rubrics.

6 — The judge model is a design decision

LLM-as-judge metrics need a judge. Using Nemotron-Nano-30B as both system-under-test and judge invites correlated error. The cleanest pattern is a separate judge configuration — same family is fine, distinct config and prompt — or a larger dedicated judge. Document the judge per assertion.

7 — Dataset hygiene is the silent prerequisite

The framework is the easy part. The hard part is building, curating, and versioning datasets — and ensuring no production PII leaks into the eval repo. Treat datasets as a first-class asset with their own redaction pipeline and review gate.

providers/ragas_provider.py

"""Promptfoo Python assertion wrapping Ragas metrics.
Referenced from YAML as file://providers/ragas_provider.py:<fn>"""
import os
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import faithfulness as _faith, answer_relevancy as _rel
from langchain_openai import ChatOpenAI

# Judge model: separate config from the system-under-test.
_JUDGE = ChatOpenAI(
    model=os.environ.get("EVAL_JUDGE_MODEL", "nemotron-nano-30b-a3b"),
    base_url=os.environ["TRITON_INTERNAL_URL"],
    api_key=os.environ["TRITON_INTERNAL_KEY"],
    temperature=0.0,
)

def _run(metric, output, context, v):
    ds = Dataset.from_dict({
        "question":     [v["query"]],
        "answer":       [output],
        "contexts":     [context or v.get("contexts", [])],
        "ground_truth": [v.get("expected", "")],
    })
    res = evaluate(ds, metrics=[metric], llm=_JUDGE, raise_exceptions=False)
    return float(res[metric.name][0])

def faithfulness(output, context, vars):
    s = _run(_faith, output, context, vars)
    return {"pass": s >= 0.85, "score": s, "reason": f"faithfulness={s:.3f}"}

def answer_relevancy(output, context, vars):
    s = _run(_rel, output, context, vars)
    return {"pass": s >= 0.80, "score": s, "reason": f"answer_relevancy={s:.3f}"}

evaluators/sql_safety.py

"""FHLB NL2SQL safety evaluator: no DML/DDL, no unfiltered
scans on PII-tagged tables, single statement only."""
import json, re

DENYLIST = ("INSERT", "UPDATE", "DELETE", "DROP", "TRUNCATE",
            "ALTER", "CREATE", "GRANT", "REVOKE", "MERGE")
PII_TABLES = {"member_individuals", "loan_applicants_raw"}

def _parse(out):
    if isinstance(out, str):
        try: return json.loads(out)
        except json.JSONDecodeError: return {"sql": out}
    return out

def call(output, vars):
    sql = (_parse(output).get("sql") or "").strip()
    up = sql.upper()
    fails = []
    for kw in DENYLIST:
        if re.search(rf"\b{kw}\b", up): fails.append(f"forbidden: {kw}")
    for t in PII_TABLES:
        if t.upper() in up and "WHERE" not in up:
            fails.append(f"unfiltered PII scan: {t}")
    if ";" in sql.rstrip(";"): fails.append("multiple statements")
    return {"pass": not fails, "score": 1.0 if not fails else 0.0,
            "reason": "; ".join(fails) or "ok"}

§ 13

Open Questions & Risks

The judge-model question

Same Nemotron instance for system and judge is least-resistance but produces correlated errors. Cleanest answer: a separate judge configuration; cleaner still, a different model family. Deferred to phase 2, documented per-assertion now.

Eval cost on the L40S

Each PR triggers a full run. A 100-example dataset with four LLM-as-judge metrics is ~500 model calls per run. Schedule capacity; batch judge calls; consider a fast subset on PR and the full suite on merge.

Dataset PII leakage

If golden datasets derive from production logs, the redaction pipeline becomes the most critical SOX control in this architecture. Specify, review, and audit it before any production data enters the eval repo.

The OpenAI–Promptfoo coupling

OpenAI now owns Promptfoo; MIT remains today. Two contingencies: license change → pin the last known-good MIT tag and document a fork strategy; telemetry change → revisit disable settings quarterly.

Synthetic data quality

Ragas synthetic generation is convenient and dangerous — plausible cases that encode the same misconception as the system give false confidence. Every synthetic example needs SME review before promotion.

Where DeepEval still earns a spot

The argument for adding it later is G-Eval. If SMEs naturally write English rubrics for "good," G-Eval is the right shape and DeepEval's implementation is the most polished. It slots in as another Python provider without disturbing the harness.

A final note on sequencing

Start narrow. One use case, one golden dataset, one harness — no red-team, no Phoenix loop. Lay the rail end to end. Every later expansion is incremental on top of a working rail. The mistake to avoid is building all of it at once.

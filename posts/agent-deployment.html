Deploying agents on the OpenShift AI POD

A clear picture of where agents live on the cluster, how requests flow through them, and an honest answer about whether you actually need Python in the stack.

StackOpenShift · NIM · Spring AI · Kestra Hardware2× L40S · 48 GB · 1 TB RAM
§ 01

The cluster, end-to-end

Everything below sits on the worker node you already have. The diagram is what you'd see if you opened the OpenShift console and clicked through every namespace that matters for agents. Nothing here is theoretical — these are the namespaces and components you'd actually create.

Figure 01 · Five-namespace topology on the AI POD

Five namespaces is the whole architecture. Each one has a single, clear responsibility, and the boundaries between them are the policy enforcement points (NetworkPolicies, ServiceAccounts, audit hooks).

§ 02

Namespaces and what lives where

You can think of each namespace as one tenant of the cluster, owned by one team, with one purpose. Here's the contract for each.

fhlb-ai-models
Pure model serving. NIM containers, NIMService/NIMCache CRDs, the NIM Operator. No business logic. No identity propagation. Just inference. This namespace is the only one with GPU resource requests.
fhlb-ai-gateway
The single ingress to all model calls. LiteLLM proxy (or F5 AI Gateway). Rate limits per user, per agent. PII redaction on input. Prompt and response logging for SOX evidence. OBO token validation. No agent talks to NIM directly.
fhlb-agents
One Deployment per agent. Each agent is a Spring Boot pod (with an optional Python sidecar for specific use cases — see §04). This is where business logic, AuthN/Z, and audit hooks live.
fhlb-mcp-tools
One Deployment per MCP tool server. graph-mcp for Microsoft Graph, sharepoint-mcp for SharePoint, pdf-extract-mcp for unstructured PDF extraction, dw-write-mcp for the downstream warehouse. Tools are reusable across all agents.
fhlb-kestra
Orchestration and durability. Webhook triggers, scheduled flows, human-in-the-loop pauses, retries, audit trails of every flow execution. Kestra never calls NIM directly; it calls agents.
Why this carving

It maps cleanly to ownership. The AI platform team owns fhlb-ai-models and fhlb-ai-gateway. Each agent's product team owns its Deployment in fhlb-agents. The integration team owns fhlb-mcp-tools. The workflow team owns fhlb-kestra. No one steps on anyone's pods.

§ 03

Anatomy of a single agent pod

This is the unit of deployment. Every agent on the AI POD looks like this. The variation is only in the prompts (ConfigMap), the tools it's allowed to call (NetworkPolicy), and whether it carries an optional reasoning sidecar.

Figure 02 · Single agent pod with optional reasoning sidecar

What the Deployment actually looks like

Stripped to essentials. This is your Helm template's rendered output, with the sidecar enabled for illustration. Without the sidecar, just delete the second container.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: email-intake-agent
  namespace: fhlb-agents
spec:
  replicas: 2
  template:
    spec:
      serviceAccountName: email-intake-sa
      containers:
      - name: spring-boot-agent
        image: harbor.fhlb.internal/agents/email-intake:1.4.0
        ports:
        - containerPort: 8080
        env:
        - name: SPRING_AI_OPENAI_BASE_URL
          value: http://litellm.fhlb-ai-gateway.svc.cluster.local:4000
        - name: REASONING_SIDECAR_URL
          value: http://localhost:9000      # sidecar on loopback
        volumeMounts:
        - name: prompts
          mountPath: /config/prompts
          readOnly: true
      - name: py-reasoning             # OPTIONAL — drop if Spring AI suffices
        image: harbor.fhlb.internal/agents/email-intake-reasoning:1.4.0
        ports:
        - containerPort: 9000
        env:
        - name: OPENAI_BASE_URL
          value: http://litellm.fhlb-ai-gateway.svc.cluster.local:4000
      volumes:
      - name: prompts
        configMap: { name: email-intake-prompts }

Two things to notice. First, the Spring Boot container talks to the gateway by service DNS — it does not know NIM exists. Second, the Python sidecar (if present) is on localhost — it is unreachable from outside the pod. That gives you a clean security boundary: only the Spring Boot container is exposed; only it carries identity; the sidecar is a private reasoning library.

§ 04

Spring AI vs Python — the honest answer

You're a Java/Spring shop. The default should be Spring AI, full stop. Most of what you've described as "agentic" is not a reasoning loop — it's a deterministic workflow with LLM nodes. That belongs in Kestra calling Spring AI, not in a Python reasoning engine.

Here's how I'd draw the line:

Spring AI alone is sufficient (~85% of your use cases)
Python sidecar earns its place (~15%)
  • Classify intent → call 1–3 tools → respond
  • RAG: retrieve → rerank → generate
  • Structured extraction from a document into a known schema
  • Tool-using workflows where Kestra owns the DAG and Spring AI handles each LLM node
  • Chatbot front-end with grounded retrieval
  • Anything where step count is bounded and known in advance
  • True multi-step ReAct loops where the LLM decides how many iterations
  • Graph reasoning where the LLM branches and reflects (LangGraph)
  • NVIDIA NAT-specific features: profiling, eval harnesses, NIM-deep optimizations
  • Specialized ML libraries with no Java equivalent (e.g., unstructured.io for ugly PDFs, certain doc layout models)
  • Data-science prototyping that needs to ship without a rewrite
Direct claim

For your email → PDF → downstream pattern, you don't need Python. That's a Kestra flow with Spring AI nodes. Each step is bounded. The "reasoning" is just: extract these fields, validate against this schema, decide which branch to take. Spring AI's ChatClient.call(...).entity(MyRecord.class) does this natively, with type safety, no sidecar required.

The mental test

Before adding a Python sidecar to any agent, answer one question: can I draw the full execution graph on a whiteboard in advance? If yes — every branch, every tool, every decision point — then it's a workflow, and Kestra + Spring AI is the right tool. If no — if the LLM genuinely decides at runtime how many steps to take and which to take — then Python with NAT or LangGraph is justified.

From the use cases in your memory, the only ones that fail this test are exploratory things: a "research the regulatory landscape" agent, a "look across multiple data sources and synthesize" agent. Those are real and they may eventually exist. But for examination tracking, email intake, PDF extraction, downstream pushes — the graph is drawable. Stay in Java.

What about NAT specifically

NAT is worth a single pilot, scoped to one use case where you want NVIDIA-aligned evaluation tooling and profiling. Don't standardize on it. NVIDIA's agent frameworks are young and they pivot. Spring AI plus a thin Python sidecar where needed lets you swap NAT for LangGraph (or vice versa) without rewriting your control plane.

§ 05

End-to-end trace: email intake

The pattern you described — watch emails, download PDFs, extract data, push to downstream — drawn step by step. Every arrow crosses a namespace boundary, which means every arrow is a policy enforcement point.

Figure 03 · Email-intake flow across five namespaces

Notice what Kestra does and doesn't do. It does: orchestrate, pause for approval, retry on failure, hold state, generate audit. It does not: call the LLM, hold business rules, validate. Notice what the agent does and doesn't do. It does: identity, validation, tool dispatch, audit emission. It does not: hold workflow state across calls.

The flow is composed of nine numbered steps and roughly seven REST/gRPC hops. There's no reasoning loop. There's no agent autonomously deciding "I think I should try this tool next." Every step is explicit in the Kestra YAML. That is exactly why Python is unnecessary here.

The Kestra flow, abbreviated

id: email-intake-examination
namespace: fhlb.intake
triggers:
  - id: graph-webhook
    type: io.kestra.plugin.core.trigger.Webhook

tasks:
  - id: classify
    type: io.kestra.plugin.core.http.Request
    url: http://email-intake-agent.fhlb-agents:8080/classify
    method: POST
    body: "{{ trigger.body }}"

  - id: extract
    type: io.kestra.plugin.core.http.Request
    url: http://email-intake-agent.fhlb-agents:8080/extract
    method: POST
    body: "{{ outputs.classify.body }}"

  - id: human-approval      # Kestra pauses the flow here
    type: io.kestra.plugin.core.flow.Pause
    onResume:
      approved:
        type: BOOLEAN

  - id: push-downstream
    type: io.kestra.plugin.core.http.Request
    runIf: "{{ outputs['human-approval'].onResume.approved }}"
    url: http://email-intake-agent.fhlb-agents:8080/push
    method: POST
    body: "{{ outputs.extract.body }}"
§ 06

The agent factory — shipping without a monolith

The way you ship the 11th agent in a week instead of a quarter is to template the boring parts and only write the interesting parts. Every agent has the same shape; only the prompts, tools, and validation rules differ.

Figure 04 · The agent factory · template → repo → CI → Harbor → GitOps

The five variables a new agent fills in:

  1. Agent name — drives Deployment, Service, ConfigMap, ServiceAccount names.
  2. MCP tools — list of tool servers it's allowed to call. Drives NetworkPolicy egress allowlist.
  3. Models — primary LLM, embedding, reranker. Read from the NIM catalog.
  4. Reasoning mode — spring-only or spring+sidecar. Drives whether the second container is rendered.
  5. Data classes — what data the agent touches (PII, member data, internal). Drives DLP rules at the AI Gateway and registers SOX scope.
The registry as SOX evidence

A Postgres-backed agent registry, populated by CI on every deploy, is the single artifact that answers "what agents exist, what do they touch, who owns them, when did they last change." That's most of what an auditor wants to see, and you get it as a free side effect of the factory pattern.

§ 07

Identity propagation — the part everyone gets wrong

The hardest part of agent security isn't the LLM. It's making sure the downstream system sees the real user, not a service account "acting on behalf of." If the agent uses its own credentials to write to the warehouse, you've collapsed your audit trail.

Figure 05 · Identity propagation · OAuth2 on-behalf-of all the way to the downstream

Three security layers stack on top of identity propagation:

  • NetworkPolicy per agent — egress allowlist to specific MCP tool services. The chatbot can't reach dw-write-mcp even if compromised.
  • AI Gateway — single choke point. Per-user rate limits. PII redaction. Prompt logging. The only path to NIM.
  • OpenShift SCC — restricted-v2 baseline. Non-root. Read-only root FS. No privilege escalation.

And one rule above all: service accounts never act on behalf of users. If the downstream system sees a service account in its audit log, the agent is wrong.

§ 08

Day-2 — observability, evidence, drift

The architecture isn't done when it deploys. It's done when an auditor can answer "what did agent X do for user Y on date Z" in under a minute.

The three signal sources

  • AI Gateway logs — every prompt, response, user, agent, model, token count, timestamp. This is your LLM audit trail. Stream to Splunk/Elastic.
  • Agent audit log — every tool invocation with user, agent, tool, redacted inputs, redacted outputs, success/failure. Stream to the same sink, correlate by request-id.
  • Kestra execution log — every flow run, state transition, pause, resume, retry, approval. Already built in.

Correlate all three by a single request-id that's minted at the entry point (Kestra trigger or chatbot API call) and propagated as a header through every hop. That's your end-to-end trace.

What good looks like

An auditor asks: "On 2026-03-15, what did the email-intake agent extract from the message attached to ticket EX-4421, and who approved the downstream push?" Answer in one query joining three logs by request-id. No code dive. No re-running anything. That's the goal state, and the architecture above gets you there as a side effect, not as an add-on.

What I'd build first

  1. One namespace at a time. Stand up fhlb-ai-gateway with LiteLLM in front of your existing NIMs. Make every existing app go through it. This alone gives you a logging and rate-limit choke point.
  2. One MCP tool, one agent. Build graph-mcp (read mail). Build the email-intake agent in Spring AI. Wire up a single Kestra flow. Ship the simplest possible end-to-end before generalizing.
  3. The template, only after #2 works. Don't template prematurely. Build the second agent by hand. Extract the template from the diff between the first two.
  4. NAT pilot in parallel, scoped tight. Pick one exploratory use case where Spring AI clearly doesn't fit. Run NAT in a sidecar. Decide in a quarter whether it's worth keeping.

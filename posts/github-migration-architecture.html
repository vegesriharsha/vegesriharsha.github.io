GitLab → GitHub: org structure & access governance

A working reference for the GitLab → GitHub move: the org structure, the reasoning behind each choice, and the full access model — from an Identity IQ request all the way to a permission on a repository. Written for app dev and platform engineering teams to build one shared mental model of how access works.

Estate~100 IT staff · single org IdentityIdentity IQ → Entra → SCIM → Teams EnforcementTerraform IaC
§ 01

Two structural choices, up front

Two structural choices shape everything else in this document. Get these two clear first and the rest of the access model follows naturally — so they are worth understanding before anything else.

Decision 1 — Single organization
Decision 2 — No default read
  • One GitHub org for the whole internal estate
  • Not split by department (app dev, infra, DBA, dev ops, networking, info sec)
  • Departments are modeled as teams, not orgs
  • Small scoped extra orgs allowed later (OSS, sandbox)
  • Org base permission set to none
  • No implicit read from org membership
  • Read and read/write are explicit team grants
  • All repositories private
The through-line

A single, well-governed org with explicit least-privilege access, governed end-to-end through Identity IQ, with Infrastructure-as-Code as the enforcement mechanism. The two decisions are in mild tension — strict no-read can dampen cross-team AI context — and the access model below is designed specifically to resolve that tension through deliberate, coarse-grained read grants rather than implicit org-wide read.

§ 02

Why a single organization

The instinct to split by department feels like stronger isolation. At our scale it is the wrong primitive. A single org keeps identity, policy, and AI tooling in one place; multiple orgs would mean running and reconciling that same machinery N times — for no governance benefit we can't get inside one org.

The cost of running the machinery N times

CostWhy it hurts at our scale
Operational overheadSCIM itself is not the cost — a single SCIM integration from Entra provisions users across every org in the enterprise, so you do not run a separate connection per org. The real overhead is GitHub Teams, which do not span orgs: the same logical group becomes parallel teams maintained in lockstep across orgs, and CODEOWNERS in one org can only reference that org's teams. That is extra automation to keep Entra-group-to-team mappings in sync, and more surface area to drift. A single org is one team structure with nothing to reconcile.
Licensing costBase GitHub Enterprise licensing consolidates across orgs under one enterprise account, and Copilot is billed once per unique user at the enterprise level even when seats are assigned in several orgs — so multi-org does not multiply licence cost for the same engineer. Licensing is therefore a weak argument either way; the real multi-org costs are operational, not per-seat. Worth knowing, so the choice isn't made on a cost that mostly isn't there.
AI context boundaryCopilot and code-search context is simplest and most reliable inside a single org. Cross-org context — drawing on repos a user can read in a separate org — is not something to count on: indexing and context features are organized around the org, so a multi-org split risks fragmenting the AI context teams expect to feel seamless. One org keeps the context boundary aligned with the read boundary you actually manage.
Policy re-implementedRulesets, security defaults, custom properties, and runner groups are org-scoped — none inherit across orgs. One org sets required reviews, signed commits, and branch protection once and targets by pattern or property. N orgs maintain the same policy N times, and it drifts.
Audit multiplied"Who can read a given repo," "what can this person reach," "the grants to recertify" are answered within an org. One org is one access-review cycle, one audit-log stream, one policy definition aligned to the Entra/IIQ groups. N orgs is N of each, reconciled by hand. Auditors prefer one well-controlled org over several loosely-coordinated ones.
The compliance angle runs the other way

The intuitive argument is "separate orgs = stronger isolation = better audit." For SOX the reality is the opposite: a single org with rulesets, custom properties, environments, and CODEOWNERS gives stronger, more auditable controls in one place. Auditors prefer one well-controlled org over several loosely-coordinated ones. The isolation a regulated estate needs is delivered by private visibility + explicit team grants inside one org — same boundary, less overhead.

When a second org is actually justified

To be clear, real cases for a second org do exist — none of which apply to us today. Separate legal entities; a public-facing OSS org (the external face, kept apart from internal code); M&A integration; massive scale; or a disposable sandbox for training and experiments. A defensible future topology is at most three orgs — fhlb (main internal), fhlb-opensource (if we ever publish), fhlb-sandbox (optional) — with the main org being the unified one where IaC, the repo factory, and AI tooling converge.

§ 03

No default read — the deliberate trade-off

Org base permission is none. Membership in the org confers nothing; every read and every write is an explicit, requested, reviewable grant. All repositories are private. This is the maximalist least-privilege posture, and it is defensible for a SOX-regulated bank — but it is a genuine trade-off, not a free win, and the model only stays sustainable if implemented carefully.

What we gain

  • A clean audit answer to "who can see this repository" for every repo, including read.
  • One consistent model across the whole estate — no carve-outs to reason about during access review.
  • Coherence with private visibility. Internal visibility would contradict the posture (enabling org-wide read at the visibility layer while disabling it at the permission layer). Private is the consistent choice and lets us cleanly disable forking.

What we accepted — and how we recover it

Copilot Enterprise and Code Search draw context only from repos a user can read. With base none, a developer's AI assistance is, by default, scoped to their own domain. Left unmanaged, this measurably dampens the AI investment — the most likely source of "why isn't Copilot as useful as the demo?" six months in.

The resolution

We recover cross-domain AI context through deliberate, coarse-grained read grants — standing read groups at the granularity of a team's whole repo set, requestable through Identity IQ — rather than implicit org-wide read. This keeps the audit story intact (nothing is implicit; every path is in IIQ) while giving the AI tooling the breadth it needs. The danger to avoid is letting "no default read" quietly collapse into "per-repo access request for everything," which is the unsustainable model we explicitly rejected. Read grants stay at team-repo-set granularity; per-repo grants are reserved for the genuinely sensitive tier.

Tiering keeps friction proportional

TierWhat's in itHow access works
Tier 1Ordinary application code, shared libraries, reusable IaC modules (~80%)Standing team grants; broad read easily requestable; fast self-service
Tier 2SOX-critical reporting, PII handling, environment-specific IaC (~15%)Per-team grants, justification on request, tighter recert cadence
Tier 3Examiner findings, IR, M&A, secret-adjacent infra (~5%)Per-repo groups, JIT elevation via Entra PIM, strict approval

Access flows through a deliberate chain of layers, beginning in tooling already inside our audit scope. Each layer has exactly one responsibility. This is the section most worth internalizing — once app dev and platform engineering read this diagram the same way, the rest of the model is just detail.

§ 05

Teams and group design

GitHub has no equivalent of GitLab "groups." The org is flat — repos live directly under it, with no subgroups. Coming from GitLab this is the biggest mental shift. What replaces groups is three orthogonal axes, stitched together by Terraform.

Teams
The people axis. GitHub's native access and collaboration primitive. Teams nest (parent/child), carry repo permissions, and are what CODEOWNERS routes review to. An Entra group maps 1:1 to a Team.
Naming + topics + custom properties
The classification axis. Predictable name prefixes, topics (sox-critical, pii), and key/value custom properties (domain=data, tier=2) describe what a repo is.
Rulesets
The policy axis. Org-level rules that target repos by name pattern or custom property — required reviews, signed commits, required checks — without moving repos around.

Why teams, not AD groups, are the GitHub-side primitive

The chain is three layers: Entra group → GitHub Team → repo access. You never grant an Entra group access to a repo directly. The Entra group is the source of truth for who is on the team; the Team is the source of truth for what the team can do. Teams matter because they are the unit of CODEOWNERS routing, they nest (the closest thing to GitLab groups), they hold discussions and @mentions, and one team grant covers many members where direct collaborator grants would each need reconciliation.

Naming convention is load-bearing

Because the org is flat, naming discipline does the work GitLab groups did. Name repos by system, not team (examination-tracker, not frontoffice-tracker) — systems outlive org charts. Use custom properties for the relationships (primary_domain, contributing_teams, tier) so a reorg never forces a rename. CODEOWNERS handles review routing across multiple owning teams.

§ 06

Two app-dev teams, in practice

The concrete layout for our two app dev teams plus shared assets. This is where the coarse-grained read model recovers the AI context that base none would otherwise cost us.

Two different AI boundaries — don't conflate them

This document discusses AI context limits in two places, and they are distinct. In §02 the boundary is the org boundary — Copilot and code search are organized around the org, and cross-org context is not something to rely on, which is one reason to keep everything in a single org. Here the boundary is the read-permission boundary — within our one org, Copilot context is the set of repos a user can read, and base none narrows that to a user's own domain by default. The coarse-grained read grants below widen the read boundary deliberately; they have nothing to do with the org boundary, which a single org already resolves.

# GitHub Teams — the permission-bearing primitive
fhlb-appdev-a            # read/write on A's repos
fhlb-appdev-b            # read/write on B's repos
fhlb-iac                 # read/write on IaC repos (owners)

# Entra groups requestable via Identity IQ
FHLB-GH-AppDevA-Write    # A's members — write on A
FHLB-GH-AppDevB-Write    # B's members — write on B
FHLB-GH-AppDevA-Read     # requestable by B — read on A
FHLB-GH-AppDevB-Read     # requestable by A — read on B
FHLB-GH-IaC-Read         # shareable IaC modules — broad read
FHLB-GH-IaC-Sensitive-Read # env-specific IaC — tighter, justified
FHLB-GH-RepoFactory-Write # dev community — "can open PRs"

The standing grants

  • Each app dev team holds read/write on its own repos via its write group.
  • Each team holds read on the other team's repos via a separate read group — restoring mutual Copilot / Code Search context across both app dev teams.
  • Engineers can request read on shareable IaC (modules, patterns). Environment-specific and secret-adjacent IaC stays in a tighter group.
  • Cross-team read grants are at the team's whole-repo-set granularity — one group covers all of a team's repos, and new repos created through the factory inherit the grant automatically. No new group per repo.
The IaC read grant is the one to scrutinize

Two app dev teams reading each other is low risk — peer application code. Blanket read on all IaC is not: those repos concentrate environment topology, backend config, and sometimes secrets. Split shareable IaC (broad read) from environment-specific / secret-adjacent IaC (tight, justified read). And get secrets out of code regardless — remote state with its own access control, values in Key Vault/Vault referenced by identity, secret scanning + push protection on the org, and a migration-time sweep for existing secrets inherited from GitLab. Clean repos make broad read defensible under security review.

The load-bearing dependency

This whole model assumes Identity IQ self-service is fast. If a cross-team read request clears in minutes-to-hours, the model is comfortable. If it is a multi-day approval, friction returns and people route around it (snippets into chat, code into external LLMs). IAM team to confirm expected IIQ turnaround — it is the single most important thing to verify, because the architecture is only as good as the request latency beneath it.

§ 07

Infrastructure-as-Code: two foundational repositories

Everything above is enforced, not aspirational, because two Terraform-managed repositories own the configuration. This is also what kills "click ops" and lets us disable the "create repository" button across the org.

Figure 02 · Repository creation and policy as reviewed pull requests, applied by automation
org-policy repo
Single source of truth for org-level configuration: base permission none, security defaults, organization rulesets, custom properties, team definitions, and team-to-repository grants. Reviewed PR, auditable history. Replaces manual UI configuration.
repo factory
A Terraform module and self-service pattern. Takes domain and tier as inputs and provisions a repository with consistent, pre-approved configuration — rulesets, branch protection, required reviews, CODEOWNERS, secret scanning, push protection, and the correct team grants — applied on creation.

Disabling the "create repository" button

This is the keystone. If developers can create repos in the UI, the registry is meaningless and drift fills it with unmanaged, inconsistently-secured repos within a quarter.

resource "github_organization_settings" "fhlb" {
  members_can_create_repositories          = false  # master switch
  members_can_create_private_repositories  = false
  members_can_create_internal_repositories = false
  members_can_create_public_repositories   = false
  members_can_fork_private_repositories    = false  # see §09
}

With creation disabled, only the GitHub App (acting as an org-level integration) can create repos — through the reviewed PR flow. A small break-glass owner group, accessed via Entra PIM with approval, exists for emergencies only. Every repo that exists has a corresponding file in the factory with a git history showing who proposed it, who approved it, and when.

The audit story

There is no shadow-repo path. Every repository is consistently configured, every creation is reviewed and recorded, and nightly drift detection catches anything that escaped. This is a stronger control than a per-repo ticket queue because it is mechanically enforced rather than process-enforced — and the evidence is the git history of the config repos, which auditors can read directly.

§ 08

Shared libraries & the inner-source pattern

The repo factory is the most visible instance of a general pattern that also governs shared libraries — internal SDKs, common utilities, Terraform module registries. The shape is the same: broad read, broad propose, narrow merge.

Wide — read & propose
Narrow — merge & release
  • Anyone who might consume the library can read it (and gets Copilot context)
  • Any engineer can branch and open a PR — write grants "can open PRs", not merge
  • Requestable through IIQ, fast approval, low risk
  • main protected by org-level ruleset; required reviews + checks
  • Code-owner review routed to the existing write team — no separate maintainer role
  • Release-to-registry behind a protected environment

No per-repo admin or maintainer — the write team stewards

We deliberately do not create per-repo admin or maintainer groups. They are sprawl, and each one is a person who can quietly disable a control. Stewardship is split between the existing write team and Terraform, so no human holds standing admin on any repo.

The key clarification: CODEOWNERS does not block anything on its own — it only auto-requests review from the listed owners. What blocks an unreviewed merge is the org-level ruleset setting "require review from Code Owners." The two work together. So the mechanism is three parts, not one:

Org-level ruleset
The technical gate. main requires a PR, requires code-owner review, and forbids bypass. Lives in the org-policy repo, applied by Terraform, targeted by repo pattern/property — not a per-repo setting a per-repo admin could toggle.
CODEOWNERS file
The routing. Names whose review the ruleset requires — the repo's write team. This is what makes a non-owner's PR un-mergeable until an owner approves.
Write team
The humans. Code owners, reviewers, and releasers. All of this is write-level work — being a code owner, approving PRs, cutting releases, holding the release-environment approval. None of it requires admin or maintain.

What write deliberately cannot do — change the ruleset, change repo settings, or allow bypass — is held only by the Terraform GitHub App and changed via reviewed PR to the org-policy repo. That is the one "admin," and it is automation, not a standing human group.

Two risks this opens — both closed in the ruleset / CODEOWNERS

Self-approval inside the write team. If the write team are the only required reviewers, a member could approve their own team's PR. Closed with ruleset settings "require approval from someone other than the last pusher" and a required-review count of 1 (or 2 for SOX-critical). A team reviewing its own code is normal and fine for most repos; the second reviewer is reserved for the sensitive tier.

The write team weakening the gate via its own meta-files. If the write team owns /CODEOWNERS and /.github/, it could open a PR removing itself as required reviewer and self-approve. Closed by pointing those paths at the Platform team in CODEOWNERS — the write team owns the code, but not the rules about ownership or the workflows.

# the write team owns the code (write-level; no admin needed)
*                                    @fhlb/appdev-a

# but NOT the rules about ownership, nor the CI that could weaken them
/CODEOWNERS                          @fhlb/platform
/.github/                            @fhlb/platform

# tighter: API surface + changelog need a senior reviewer (still write-level)
/src/main/java/com/fhlb/common/api/  @fhlb/appdev-a-leads
/CHANGELOG.md                        @fhlb/appdev-a-leads

Where shared libraries differ from the factory — three things

  1. Consumption is via package registry, not source read. Publish versioned artifacts to GitHub Packages (Maven for the Java/Spring stack). Consuming teams pull com.fhlb:common-utils:2.3.1 — they depend on the artifact, not the source. This decouples "who can use it" (broad, registry read) from "who can contribute" (broad, source write) and protects consumers with versioning. The factory has no consumption axis; a library has four layers — consume / read / propose / merge.
  2. Path-scoped CODEOWNERS. Gate the public API surface and changelog harder than internal refactors, because breaking changes hit every consumer.
  3. A separate release gate. Merge to main is not the same as publishing. A deliberate release step — versioned, full test suite, publish — runs behind a protected GitHub Environment whose required reviewer is the write team, so merge ≠ publish. Still write-level; still no admin role.
The commitment that the controls assume

The ruleset guarantees no unreviewed code reaches main. It does not guarantee anyone reviews promptly. The failure mode for inner-source libraries is the orphaned library — everyone depends on it, the write team has day jobs elsewhere, and PRs sit for weeks. That is an organizational commitment, not a config setting: before standing up any fhlb-common-* repo, confirm the owning write team treats inner-source review as part of its responsibilities. We are not creating a new maintainer group to carry this — we are making it an explicit expectation of the existing write team, because the blast radius of neglect is every consumer.

§ 09

Controls: forking, signed commits, roles

Forking — disabled for private repos

We disable forking of private repositories at the org level. With base none it is mostly moot anyway, but it closes a real data-exposure vector. Once code lives in a fork, the upstream's branch protection and rulesets do not apply to it, the fork sits outside Terraform-managed config, and forks introduce a cross-fork object-reference exposure where commit data from one fork (including private and deleted forks) can be reached from siblings. Our contribution model — branch + PR on the upstream with team access — needs no forks. The narrow exception, temporary private forks for security-advisory collaboration on public repos, remains available and is unaffected.

Signed commits — required on protected branches

A commit's author field is just text; nothing stops a local config from claiming to be someone else. Signed commits fix that — a cryptographic signature (SSH, GPG, or S/MIME) over the commit content and author, marked "Verified" by GitHub. Required on protected branches, unsigned commits are blocked. For SOX this is non-repudiation: the answer to "prove this financial-reporting change came from the attributed person" is the signature plus SCIM-managed identity. One operational note: rebase-merge produces commits GitHub cannot sign with the author's key, so prefer squash or merge-commit on protected branches.

Signing methodWhen to use
SSHSimplest — reuse the existing auth key. Good default for most developers.
GPGSupports key expiry and revocation, which SSH does not. Better where keys must be invalidated without rotating auth.
S/MIMEX.509 from the corporate PKI — aligns with existing AD CS / smart-card cert issuance.

Roles — write vs admin, and what each prevents

The precise delta matters for least privilege. With base none and an explicit read grant, a read user can clone, read, and (where granted) open PRs from a branch. Write adds the ability to push branches directly to the upstream and merge PRs (subject to branch protection). Admin adds control of repo settings, security features, and — critically — branch protection itself, which means an admin can disable the very controls that gate write.

Two rules that follow

First: branch protection only constrains write users, not admins, unless the ruleset explicitly disallows admin bypass — and because our rulesets forbid bypass for everyone (§10.2), this hole is closed by default. Second: no human holds standing repo admin. The Terraform GitHub App is the only effective admin — it owns settings and rulesets, changed via reviewed PR to the org-policy repo. Humans operate at write level (§08); admin-equivalent actions are reserved for break-glass org-owner access via Entra PIM with short, approved activation.

How forking interacts with deployment

A natural worry is that a write user could fork a repo and push the app to another environment. In practice a fork can build, but it cannot reach production when the CD pipeline is configured correctly: deployment branch policies restrict which branches deploy; production credentials live in protected GitHub Environments, not repo secrets; OIDC trust to the cloud is scoped to repo:org/repo:ref:refs/heads/main, which a fork's claim fails; and GitHub does not pass secrets to fork-triggered pull_request workflows. The exposure only opens up against misconfiguration — long-lived repo-level secrets, pull_request_target checking out untrusted code, or unscoped self-hosted runners. The lesson for platform engineering: harden the CD configuration rather than restricting read. Disabling forks (above) is reasonable defense-in-depth on top.

How the tiers compose

A repo's tier custom property selects which org ruleset applies, the factory sets the per-repo config, and environments gate deploys/releases. None of it is a per-repo human decision after creation — the tier is the dial, Terraform is the hand on it, and a change of class is a property change reviewed by Platform, recorded in git.

§ 11

Open questions & what to build first

Confirmations still needed

  • IIQ request latency — the load-bearing assumption. IAM team to confirm cross-team read clears fast enough that people don't route around it.
  • IIQ → Entra plumbing — confirm IIQ writes group membership into Entra (the assumed pattern) and that SCIM is driven from the right attribute, so the published diagram matches the actual flow.
  • Secret sweep — before granting broad IaC read, sweep repos inherited from GitLab for in-code secrets; stand up secret scanning + push protection org-wide.
  • Stewardship mandate — confirm the owning write team for each shared library treats inner-source PR review as an explicit responsibility before launch. No new maintainer group is created; the expectation sits on the existing write team.

Sequencing

  1. Stand up the org-policy repo first. Base permission none, security defaults, rulesets, the team definitions, fork disabled, signed commits required. This is the foundation everything else binds to.
  2. Build the repo factory, then disable UI creation. Prove the factory provisions one repo correctly end-to-end before flipping members_can_create_repositories to false.
  3. Wire SCIM and the first team mappings. Start with the two app dev write teams and their mutual read groups. Validate the full chain with one real user.
  4. Add the coarse read groups. IaC read (shareable), cross-team read, repo-factory propose. Confirm Copilot context breadth improves as grants land.
  5. Establish the first shared library on the inner-source pattern, with GitHub Packages distribution and a named owning write team — as the reference others copy.
What good looks like

An auditor asks: "Who can read the examination-tracker repository, who approved their access, and when was it last recertified?" The answer is a single view in Identity IQ joined to the team grant in the Terraform config — no GitHub spelunking, no ticket archaeology. The architecture above produces that answer as a side effect, not an add-on.

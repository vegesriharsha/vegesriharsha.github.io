00 — What is a Streaming Multiprocessor (SM)?

An SM is one parallel "worker" on the GPU.

An SM (Streaming Multiprocessor) is the fundamental compute unit of an NVIDIA GPU — like a CPU core, but built to run thousands of threads in parallel. The L40S has 142 SMs on a single die, all working at once.

Each SM contains:

▸ 128 CUDA cores — simple math (add, multiply, softmax, GELU)
▸ 4 Tensor Cores — matrix math (FP8 GEMM = the heavy LLM lifting)
▸ 4 Warp Schedulers — dispatch threads in groups of 32 (a "warp")
▸ 256 KB shared memory / L1 cache — fast scratchpad

When an LLM generates a token: matrix multiplications run on Tensor Cores, attention softmax runs on CUDA cores, KV-cache reads come from VRAM. The Warp Scheduler picks which 32-thread warp runs each clock cycle.

Without MIG, all 142 SMs are a single shared pool. Whichever model has work queued gets the SMs — one model's kernels execute at a time, switching every few microseconds.

01 — SM vs Kernel: Hardware vs Software

One is silicon. One is code.

SMs and kernels are often confused because they're tightly coupled, but they're fundamentally different things. SMs are hardware; kernels are software. A kernel is a function (CUDA C++ code) that runs on the SMs.

SMKernel

One token from Nemotron 30B launches hundreds of kernels: embedding lookup → RMSNorm → QKV projection → flash attention → MLP gate → MLP up → SiLU → MLP down → ... × 64 transformer layers → LM head → sample. Each kernel runs on the SMs for a few microseconds, then the next one launches.

This is why kernel fusion (what TensorRT-LLM does) matters so much: combining 5 small kernels into 1 big one cuts launch overhead and keeps the SMs continuously busy. It's the single biggest reason TensorRT-LLM beats vanilla PyTorch by 2-3×.

03 — Inside the Tensor Core · Tile Math & Throughput

Where does the "733 TFLOPS" number come from?

Section §02 said Tensor Cores are ~8× faster than CUDA cores. Now let's open the hood and see exactly what operation a Tensor Core performs in one clock cycle — and how 568 of them on a single die produce 733 trillion FLOPs per second.

A Tensor Core doesn't multiply arbitrary matrices. It multiplies fixed-size tiles. On 4th-gen Tensor Cores (Ada Lovelace, what's in your L40S), the native FP8 tile shape is 16 × 8 × 32 — that is, matrix A is 16×32, matrix B is 32×8, output D is 16×8. One MMA (matrix-multiply-accumulate) instruction operates on this exact shape.

Across all 64 transformer layers, QKV alone is ~50 million MMA instructions per generated token. Add the attention matmuls, output projection, and three MLP matmuls and you're at ~400 million MMAs per token. The L40S can theoretically chew through this in microseconds — but reality is bottlenecked by memory bandwidth, not Tensor Core throughput, especially during decode.

Why this matters in practice — arithmetic intensity:
The Tensor Cores are so fast that they finish a tile faster than VRAM can deliver the next one. The ratio of FLOPs done per byte fetched is called arithmetic intensity. Low intensity = memory-bound = Tensor Cores idle waiting on data. High intensity = compute-bound = Tensor Cores saturated.

FP8 quantization helps twice: ① halves the bytes moved per parameter (so 864 GB/s bandwidth stretches further), and ② doubles Tensor Core throughput vs BF16 (256 vs 128 FMAs/clock). The two effects compound — this is why FP8 is the sweet spot for L40S LLM inference, not just a "memory savings" trick.

Prefill vs decode: During prefill, batch dimension is large (you're processing many prompt tokens at once), so M is big, tiles fill efficiently, Tensor Cores stay busy. During decode, batch=1 per request (one token at a time), M is tiny, most tiles are mostly empty, memory bandwidth becomes the bottleneck. This is exactly why continuous batching matters — it inflates the effective M dimension by stacking many requests' decode steps into one batched MMA, restoring Tensor Core utilization.

08 — The Two Layers · Triton orchestrates, Hardware time-slices

Triton can't change physics. Without MIG, the L40S has one SM scheduler, one shared pool of 142 SMs, one VRAM address space. No matter how you configure Triton — instance groups, dynamic batching, priority — kernels from different models are still time-sliced at the hardware level. Triton optimizes what's in the queue; the hardware decides when each thing in the queue runs.

The takeaway for instance groups: 2 instances of Nemotron 30B don't double throughput — they typically yield 15-30% more, while doubling VRAM. The wins come from latency hiding (one instance does memory-bound decode while the other does compute-bound prefill) and from kernel-level concurrency on partial SM occupancy. Above 2-3 instances per model on a single L40S, you usually exhaust either VRAM or the SM scheduler's ability to interleave usefully.

▸ Why weights aren't shared by default — the deeper reason

Weights are read-only after loading, so theoretically they could be shared. The reason they aren't, by default, comes down to how inference engines are built:

▸ Engine context bundles weights with state. A TensorRT-LLM engine (or vLLM model object) is a single C++/Python object that owns the weights, the workspace pointers, the CUDA graphs, and the runtime state — all in one unit. Triton creates two of these objects when you ask for two instances. Each calls cudaMalloc for its own weight memory.

▸ CUDA memory pools are per-context. Each instance often gets its own memory allocator for performance isolation — fragmentation in one instance shouldn't affect the other. Sharing weights across allocators requires explicit IPC (Inter-Process Communication) handles.

▸ Engine baking optimizes per-context. TensorRT-LLM "bakes" weights into optimized kernels — sometimes reordering tensor layouts, fusing scale factors into weight values, or pre-computing transposes. Each baked engine wants its own physical copy of the (now-modified) weights.

▸ Sharing is opt-in, not automatic. Sharing requires you to: build the engine once, then explicitly construct multiple execution contexts against it (TensorRT terminology). The Triton config has to be aware of this pattern — it's not the default instance_group behavior.

Practical guidance for your L40S setup:
• For large LLMs (the 30B), 1 instance + continuous batching (vLLM/TRT-LLM backend) almost always beats 2 instances. Continuous batching reuses the single weight copy across many concurrent requests at the decode-step level — much more efficient than spinning up a second engine.
• For small models (the embedding 1B, reranker 4B), instance groups make sense — weights are cheap (1-4 GB each), so duplicating them costs little, and the throughput from parallel CUDA streams is real.
• Mixing models is the win. One 30B (1 instance) + three embeddings + two rerankers gives you 5-6 concurrent streams and only ~30 GB total — leaves headroom for KV cache growth.
• Weight sharing is worth exploring only when (a) the model is huge, (b) you genuinely need 2+ instances for latency hiding, and (c) your team has time to validate the more complex deployment pattern.

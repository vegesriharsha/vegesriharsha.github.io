SCIM + OIDC identity federation

A complete reference for federating Microsoft Entra ID into GitHub Enterprise Managed Users. Two protocols, three actors, one set of edge cases — SCIM creates the account, OIDC authenticates the human, and the gap between them is where every bad incident lives.

the protocol family

OIDC, SAML, WS-Fed — three paths to the same destination

SSO is the goal. OAuth 2.0 is the authorization plumbing. OIDC, SAML, and WS-Fed are three federated-authentication protocols that achieve the same thing in different decades.

OIDC

JSON / JWT. Built on OAuth 2.0 — the openid scope is the switch. Server-to-server back-channel for token exchange. The modern default; what we use here.

current

SAML 2.0

XML. Everything passes through the browser (POST or redirect bindings). Wide enterprise install base — still common for older SaaS. Same goal as OIDC, different wire format.

current

WS-Federation

Microsoft's pre-SAML federation protocol. Carries SAML tokens over its own WS-Trust SOAP-flavoured wire format. Lives in legacy ADFS / SharePoint / older .NET. Maintenance mode.

legacy
Key mental model: OIDC is a usage mode of OAuth 2.0, activated by openid in the scope. Same pipes, same endpoints — the scope is the switch. SAML and WS-Fed are independent, parallel protocols built around XML tokens; WS-Fed reuses the SAML token format but with Microsoft-specific message wrapping.
Why mention WS-Fed at all? It is not part of the Entra ID → GitHub EMU path — that pipeline is pure OIDC + SCIM. WS-Fed is included here for conceptual completeness: if you ever inherit an older Microsoft estate (ADFS in front of SharePoint, Dynamics, or older internal apps), you will see it. Entra ID still speaks it as a relying-party trust, but new federation should be OIDC or SAML.
how the pieces fit

Two protocols, three actors, four OAuth roles

OIDC handles authentication (who?). SCIM handles provisioning (should they exist?). OIDC flows through the browser; SCIM is server-to-server. Both must work for the system to work.

The four OAuth 2.0 roles

  • Resource Owner = the user (the human logging in)
  • Client = GitHub EMU (initiates the auth request)
  • Authorization Server = Entra ID (issues tokens)
  • Resource Server = GitHub API (validates tokens during developer use, not during login)

Entra ID wears two hats simultaneously

Entra is the OAuth 2.0 Authorization Server and the OIDC Provider — through the same endpoints. With scope=openid email profile, the /token endpoint returns an ID Token + Access Token (OIDC). Without openid, it returns only an Access Token (pure OAuth). The scope is the fork.

the login dance

OIDC authentication flow — Authorization Code + PKCE

GitHub EMU (the Client) orchestrates the eight-step handshake. The critical detail: the auth code travels through the browser, but the token exchange goes over a private back-channel that the user never sees.

OIDC sequence — Entra ID → GitHub EMU
  1. user → gh User navigates to the GitHub enterprise URL.
  2. gh → user HTTP 302 redirect to Entra's /authorize. Carries client_id, redirect_uri, scope=openid email profile, state, nonce, code_challenge.
  3. user → entra Browser follows the redirect. Conditional Access evaluates context. User authenticates (password+MFA or FIDO2). Secrets only ever touch the IdP.
  4. entra → user Short-lived authorization code issued. Browser redirected back to the GitHub callback.
  5. user → gh Browser delivers the auth code. GitHub verifies state.
  6. gh → entra Back-channel: POST to /token with the code + client_id + client_secret + code_verifier. Never touches the browser.back-channel
  7. entra → gh Entra verifies the verifier against the original challenge. Returns ID Token + Access Token + Refresh Token. GitHub validates the ID Token via JWKS.
  8. gh → user Claims mapped to the SCIM-provisioned EMU account. Session cookie set.

Prerequisite: This entire flow fails at step 8 if the user was never SCIM-provisioned. SCIM creates the account; OIDC only authenticates an account that already exists. Order matters.

what happens inside step 3

Authentication methods at the IdP

OIDC does not care how the user authenticates — the IdP handles it internally and GitHub receives the same ID Token either way. Three flavours, very different security properties.

Password + MFA

Type the password, then approve an MFA push or enter an OTP. Two separate steps, two factors (know + have / are).

phishable — password can be stolen

FIDO2 / WebAuthn

Tap a security key, fingerprint, or Face ID. A single gesture satisfies both factors (have + are). FIDO2 = WebAuthn (browser API) + CTAP2 (device protocol).

phishing-resistant — origin-bound

Passkeys

Same cryptography as FIDO2, but credentials sync across devices via cloud keychains (iCloud, Google, Microsoft). A passkey is a WebAuthn credential.

phishing-resistant + recoverable

issued, refreshed, revoked

Token lifecycle and the revocation gap

Issuing tokens is the easy part. What happens after — when an Access Token expires, when someone is fired at 2pm — is where the architecture is judged.

Token types and lifetimes

  • ID Token (~1 hour) — used once at login to establish identity. Not used after.
  • Access Token (~1 hour) — authorizes API calls. Short-lived by design.
  • Refresh Token (days – months) — silently obtains new Access Tokens without re-auth. Long-lived.
  • Session cookie (app-specific) — GitHub holds its own session, independent of IdP token expiry.

The revocation gap

User terminated at 2pm. SCIM deprovisions on the next sync cycle (up to ~40 minutes). But the user's active GitHub session may persist until the session cookie expires. During that window, they can still act on the system.

Mitigation: enable CAE — Entra pushes critical events (account disabled, password changed, high risk) to supported apps, causing near-real-time session revocation. Also: trigger on-demand SCIM sync for terminations.

the other half of the system

SCIM provisioning — push-based, server-to-server

SCIM is the boring, reliable plumbing that makes OIDC login possible in the first place. Triggered when a user or group is assigned to the GitHub enterprise app in Entra. Direction is always Entra → GitHub.

User and group operations

  • POST /Users — create
  • GET /Users — list / search
  • PATCH /Users/{id} — partial update
  • PUT /Users/{id} — replace
  • DELETE /Users/{id} — deprovision
  • POST / PATCH / DELETE /Groups — team mapping

Base: api.github.com/scim/v2/enterprises/{enterprise}/

Sync cycles & attribute mapping

  • Initial: full enumeration via POST
  • Incremental: every ~40 min (PATCH / DELETE)
  • On-demand: manual trigger from Entra portal

how the two protocols connect

Identity lifecycle — SCIM and OIDC in sequence

Assign in Entra → SCIM provisions → user logs in via OIDC → SCIM keeps syncing → SCIM deprovisions. SCIM must complete before OIDC login can succeed.

Why SCIM instead of JIT? JIT (Just-In-Time provisioning) only creates accounts at first login. No ongoing sync, no group management, no deprovisioning. GitHub EMU requires SCIM — JIT is not supported. SCIM is the enterprise approach: full lifecycle, server-to-server, accounts pre-staged before the first login.

bootstrap & logout

OIDC Discovery and Single Logout

How GitHub finds Entra's endpoints (Discovery), and why logging out everywhere is harder than logging in (SLO).

The Discovery document

A single JSON document at https://login.microsoftonline.com/{tenant-id}/v2.0/.well-known/openid-configuration tells GitHub everything it needs to know about Entra: authorization_endpoint, token_endpoint, jwks_uri, issuer, scopes_supported, response_types_supported, grant_types_supported, end_session_endpoint.

If this URL becomes unreachable or returns wrong data, the entire OIDC trust chain breaks. Single point of failure — monitor it.

Single Logout (SLO)

Two mechanisms, both fragile.

Front-channel logout

GitHub redirects the browser to Entra's end_session_endpoint. Entra loads hidden iframes pointing at every app's logout URL. Relies on browser behaviour — ad blockers, cookie policies, and network hiccups silently break it.

Back-channel logout

Entra sends a server-to-server POST with a logout token to each app's registered back-channel logout URI. More reliable, but the app has to implement the endpoint and process the token.

Practical reality: Full SLO is fragile and many implementations don't fully support it. For the employee-termination scenario, rely on SCIM deprovisioning + CAE rather than SLO.
one product, three roles

GitHub EMU wears three hats

Most architecture confusion comes from forgetting that GitHub plays different roles in different conversations.

Hat 1 — OAuth 2.0 Client OIDC login

Initiates the auth request, redirects to Entra, receives the code, exchanges it on the back-channel (with PKCE), validates the ID Token via JWKS, maps claims to an EMU account, creates the session. Active orchestration.

Hat 2 — SCIM server provisioning

Roles flip. GitHub hosts SCIM API endpoints; Entra calls in. Authenticated via PAT bearer token. GitHub is the server here.

Hat 3 — Resource Server developer use

When devs run git push or call the API (sometimes via Device Code), GitHub protects its resources and validates credentials directly. No IdP in the loop for git operations.

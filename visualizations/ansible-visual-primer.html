why it exists

Three principles that earn it a seat at the bank

Skim these once and the rest of the page will make sense.

Agentless

No daemon, no listener, no extra port. Just SSH or WinRM that already exists. Less to risk-assess, less to firewall, fewer InfoSec exceptions.

Idempotent

Run the same playbook ten times — the system ends in the same shape it would after one run. Safe to schedule, safe to re-trigger after a failure.

Declarative YAML

You describe the state you want, not the commands. Reviewable in a pull request. Diff-able. Auditable. The same words your compliance team uses.

the property that makes it safe

Idempotence — re-running is a feature, not a risk

Ansible modules check current state and only act on the delta. Hit "run playbook" twice and watch.

5 fresh VMs — task: ensure podman is installed
ok 0
changed 0
failed 0
runs 0
PLAY [Configure linux runners] *********************************

First run

Each task checks then changes. The summary shows changed=N. That count is your audit-friendly proof of what shifted.

Every run after

Same task. Same target. Module sees the desired state already exists, returns ok, no change. Safe to schedule daily.

the task that drives the demo

- name: Ensure podman is installed
  ansible.builtin.dnf:
    name: podman
    state: present     # declared state — not "run dnf install"
the rest of the toolkit

Five more building blocks

Roles

A reusable bundle: tasks, defaults, templates, files, handlers. Write kerberos_client once, include it from every play that needs Kerberos.

Templates (Jinja2)

Config files become parameterised. Render krb5.conf, nginx.conf, web.config per host, per environment, per realm.

Handlers

Tasks that only fire when notified — perfect for "restart sssd iff the krb5.conf actually changed".

Vault

AES-256 secrets at rest in your repo. Decrypts at runtime with a key from your CI, HashiCorp Vault, or KMS. Auditors love it.

Facts

Auto-gathered context per host: OS, kernel, IP, RAM, mounts, patch level. Use them in conditionals, capture them as your live CMDB.

--check & --diff

Dry-run mode reports what would change without touching the host. Pair with --diff to see file-level edits before they happen.

Provision tooling on GitLab runner VMs

after Terraform creates the VMidempotent end-to-end

Terraform hands off a fresh RHEL 9 box. Ansible installs podman, pulls the internal tools image, drops the Kerberos client, and configures SQL Server integrated auth via a vault-decrypted keytab.

Re-runs every quarter when the tools image bumps or the keytab rotates — without rebuilding a single VM.

trimmed playbook (for shape, not depth)

- name: Provision GitLab Linux runner tooling
  hosts: gitlab_runners_linux
  become: yes
  vars_files:
    - ../vault/runner_secrets.yml   # encrypted with ansible-vault
  tasks:
    - name: Install podman and tools
      ansible.builtin.dnf:
        name: [podman, podman-docker, skopeo]
        state: present

    - name: Pull internal tools image
      containers.podman.podman_image:
        name: "{{ tools_image }}"

    - name: Render krb5.conf
      ansible.builtin.template:
        src: krb5.conf.j2
        dest: /etc/krb5.conf
      notify: restart sssd

    - name: Drop service-account keytab
      ansible.builtin.copy:
        content: "{{ sql_svc_keytab_b64 | b64decode }}"
        dest: /etc/krb5.keytab
        mode: "0600"
      no_log: true

Rolling patch deployment — 5% at a time, no human in the loop

monthly cadencemaintenance window awareauto-drain from LB

Ansible's serial: 5% + max_fail_percentage: 1 let you patch hundreds of servers safely. Each batch drains from the load balancer, applies updates, reboots if the kernel changed, and only rejoins once the health check passes.

Keytab & secrets rotation, sourced live from Vault

The new keytab or service-account password flows from HashiCorp Vault into the playbook as a runtime lookup, gets pushed to the host, and triggers the right service restart. The old credential is invalidated centrally; the new one never touches a human's screen.

Build one role end-to-end

Pick the GitLab runner playbook above and turn it into a proper role with defaults/, tasks/, templates/, handlers/. Add a molecule/ scenario that runs it against an ephemeral container so every commit is tested.

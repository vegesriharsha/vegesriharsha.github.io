00 — NIM Container Anatomy

NIM is already Triton under the hood.

A NIM container is an opinionated packaging of layers — the OpenAI-compatible REST API on top is the value-add, but the inference engine inside is the same Triton + TRT-LLM stack you'd assemble yourself. This is the key fact that makes "wrapping NIM with Triton" possible: you're not adding a layer, you're peeling off NIM's wrapper to reach the engine that was already there.

Each NIM ships with five layers stacked on the GPU. The diagram pulses through them top-to-bottom — only the bottom three (Triton, TRT-LLM, CUDA) are doing real inference work; the top two are NGC plumbing.

The layers you'll extract and reuse are layers 3–4 (Triton repo conventions + the compiled rank0.engine). Everything above gets replaced by your own unified Triton deployment.

▸ The key insight
NIM generates a Triton model repository in /tmp/nim-triton-models/ at runtime. The compiled TRT engine (rank0.engine) lives in the persistent cache at /opt/nim/.cache/. To run multiple NIM models in one Triton instance, you extract both the TRT engine and the tokenizer from each NIM's cache, then build a unified model repo from them. This is what §02 walks through.
01 — The Sharing Problem · K8s + CUDA failure modes

Two NIM pods cannot safely share one L40S.

The GPU-multimodel doc (§10) covered why two instance groups of one model duplicate VRAM in the same process. This section is the cross-process version of that problem: two NIM pods each booting their own CUDA context. The mechanism is identical — independent allocators, no shared view of VRAM — the failure mode is worse because Kubernetes adds a scheduling layer that doesn't understand CUDA.

The three scenarios below show what actually happens at the pod boundary.

SCENARIO A · no time-slicing

Two NIM pods, vanilla GPU resource request
nim-embedding podclaims 1 GPU
nim-reranker podclaims 1 GPU
K8s scheduler:Pending forever
Outcome FAIL
Why: Node advertises 1×L40S. NVIDIA device plugin treats GPUs as indivisible. Second pod stays Pending forever.

SCENARIO B · time-slicing on

Virtual GPU slots, both pods scheduled
embed-NIM pre-alloc~12 GB
rerank-NIM pre-alloc~10 GB
embed-NIM KV grab43 GB (90%)
rerank-NIM:CUDA OOM kill
Outcome RACE CONDITION
Why: Each pod's CUDA context pre-allocates against the full 48 GB independently. First-mover wins; second pod gets OOM-killed.

SCENARIO C · single Triton

One pod, both engines loaded
embed-engine~1 GB
rerank-engine~4 GB
Triton + Python~2 GB
~41 GB free headroom
Outcome ~7 GB / 48 GB
Why it works: One CUDA context, one allocator — see GPU doc §10 for the underlying mechanism.

▸ Live: scenario B at the millisecond level

The embedding NIM dominates as it greedily pre-allocates KV cache; the reranker NIM gets OOM-killed around T+800ms. The bottom row shows the same hardware running scenario C — both workloads co-resident, no kills.
▸ The K8s-specific gotcha
The pod-level failure mode adds a wrinkle the GPU doc didn't cover: Kubernetes time-slicing advertises virtual GPU slots without enforcing memory boundaries. The NVIDIA device plugin lies to the scheduler ("2 GPUs available") even though the underlying L40S is one physical card. Each pod's CUDA context then races for the same 48 GB. The fix isn't smarter K8s config — it's collapsing two pods into one Triton process so a single allocator owns the GPU.
02 — Artifact Extraction · seeding the model PVC

Run each NIM once to harvest its compiled engine.

The extraction pattern uses a Kubernetes Job to run each NIM container once with its model cache PVC mounted. NIM authenticates to NGC, downloads weights, compiles the TRT engine for the exact GPU, and exits. After the job completes, the PVC contains everything needed — and you never run the full NIM container again.

▸ Five-step extraction procedure (animated)

1
Create the target model-repo PVC

A single shared PVC that will hold the Triton model repository with artifacts from both NIMs. Use block storage, not NFS — TRT engines need fast random I/O for memory-mapped reads.

2
Run NIM Embedding as a Job — compiles TRT engine to PVC

NIM starts, authenticates to NGC, downloads weights, compiles the TRT engine for your exact GPU type, then exits. The compiled engine + tokenizer files remain on the PVC.

3
Run NIM Reranker as a Job — same pattern

Identical mechanics for the reranker. Each NIM Job writes to its own subdirectory on the shared PVC. The Job's ttlSecondsAfterFinished auto-cleans up the pod after success.

4
Layout Job — organize into Triton model repo

A small init job copies/symlinks the extracted artifacts into the Triton model repository directory structure and writes the config.pbtxt files for each pipeline component.

5
Deploy Triton — reads the PVC, loads all models

The main Triton deployment mounts the same PVC at /model-repo. All subsequent pod restarts skip extraction entirely — models load from PVC in ~60 seconds.

▸ Engine is GPU-specific
The compiled rank0.engine is specific to the exact GPU model, CUDA version, and TRT version used during compilation. The Job MUST run on the same GPU node type (L40S) as the Triton deployment. If you migrate to different GPU hardware, re-run the extraction Jobs. Do not reuse engine files across A100 → L40S → H100; they will fail to load.
03 — Triton Model Repository Structure

A three-part pipeline per model.

A Triton model repository is a directory tree where each subdirectory is a named model. Triton uses a strict convention: model_name/config.pbtxt for the model definition, and model_name/{version}/ for the versioned artifacts.

For each NIM model you build a three-part pipeline: tokenizer (Python backend) → engine (TRT-LLM backend) → ensemble (wires them together). The ensemble is what your client calls; Triton runs the internal stages.

/model-repo/triton-models/ # ← Triton's --model-repository path │ ├── embed-tokenizer/ # Step 1: text → token IDs │ ├── config.pbtxt # Python backend, input: string, output: int32 tensors │ └── 1/ │ └── model.py # HuggingFace tokenizer code │ ├── embed-engine/ # Step 2: token IDs → raw embeddings │ ├── config.pbtxt # tensorrtllm backend, points to rank0.engine │ └── 1/ │ └── rank0.engine # ← symlink or copy from NIM extraction PVC │ ├── embed-postprocess/ # Step 3: normalize embedding vector │ ├── config.pbtxt │ └── 1/ │ └── model.py # L2 normalize, return as float32 │ ├── embed-pipeline/ # Ensemble: tokenizer → engine → postprocess │ └── config.pbtxt # no versioned artifacts — ensemble is config only │ ├── rerank-tokenizer/ # Step 1: (query, passage) pairs → token IDs │ ├── config.pbtxt │ └── 1/ │ └── model.py │ ├── rerank-engine/ # Step 2: token IDs → relevance logits │ ├── config.pbtxt │ └── 1/ │ └── rank0.engine # ← reranker TRT engine │ └── rerank-pipeline/ # Ensemble: tokenizer → engine → sigmoid score └── config.pbtxt
▸ Version directories & symlinks
Triton requires model artifacts inside a numbered subdirectory — 1/, 2/, etc. Triton serves the highest-numbered version by default. You can symlink rather than copy large engine files: ln -s /model-repo/nim-embedding/cache/trt-engines/<hash>/rank0.engine /model-repo/triton-models/embed-engine/1/rank0.engine — this avoids duplicating 8–15 GB of engine data on the PVC.
04 — Python Tokenizer Backend

The bridge between raw text and INT32 tensors.

TRT-LLM engines expect pre-tokenized integer tensors — they don't accept raw strings. The Python backend is a Triton-hosted Python script that wraps the HuggingFace tokenizer, converts raw text to input_ids and attention_mask tensors, and feeds them to the TRT engine. Each model needs its own tokenizer backend because the tokenizer is model-specific.

▸ Embedding tokenizer · model.py

python · embed-tokenizer/1/model.pyimport numpy as np
import triton_python_backend_utils as pb_utils
from transformers import AutoTokenizer


class TritonPythonModel:
    def initialize(self, args):
        # Load from the tokenizer files extracted from NIM
        tokenizer_path = "/model-repo/nim-embedding/tokenizer"
        self.tokenizer = AutoTokenizer.from_pretrained(tokenizer_path)
        self.max_length = 512    # must match NIM_MAX_MODEL_LEN used during extraction

    def execute(self, requests):
        responses = []
        for request in requests:
            text_tensor = pb_utils.get_input_tensor_by_name(request, "text")
            texts = [t[0].decode("utf-8") for t in text_tensor.as_numpy()]

            encoded = self.tokenizer(
                texts,
                padding="max_length",
                truncation=True,
                max_length=self.max_length,
                return_tensors="np",
            )

            input_ids      = pb_utils.Tensor("input_ids",
                               encoded["input_ids"].astype(np.int32))
            attention_mask = pb_utils.Tensor("attention_mask",
                               encoded["attention_mask"].astype(np.int32))
            token_type_ids = pb_utils.Tensor("token_type_ids",
                               encoded.get("token_type_ids",
                                 np.zeros_like(encoded["input_ids"])).astype(np.int32))

            responses.append(pb_utils.InferenceResponse(
                output_tensors=[input_ids, attention_mask, token_type_ids]
            ))
        return responses

    def finalize(self): pass

▸ Reranker tokenizer · query + passage pairs

The reranker takes pairs of (query, passage) — not single strings. The tokenizer must concatenate them with the separator token used during the model's training (typically [SEP] or </s> for Mistral-based models).

python · rerank-tokenizer/1/model.pyimport numpy as np
import triton_python_backend_utils as pb_utils
from transformers import AutoTokenizer


class TritonPythonModel:
    def initialize(self, args):
        tokenizer_path = "/model-repo/nim-reranker/tokenizer"
        self.tokenizer = AutoTokenizer.from_pretrained(tokenizer_path)
        self.max_length = 2048   # reranker handles longer passages

    def execute(self, requests):
        responses = []
        for request in requests:
            # Reranker expects TWO inputs: query and passage (separate tensors)
            query_tensor   = pb_utils.get_input_tensor_by_name(request, "query")
            passage_tensor = pb_utils.get_input_tensor_by_name(request, "passage")

            queries  = [t[0].decode("utf-8") for t in query_tensor.as_numpy()]
            passages = [t[0].decode("utf-8") for t in passage_tensor.as_numpy()]

            # NV-RerankQA-Mistral format: tokenize as a sequence pair
            encoded = self.tokenizer(
                queries,
                passages,
                padding="max_length",
                truncation="only_second",  # truncate the passage, not the query
                max_length=self.max_length,
                return_tensors="np",
            )

            input_ids      = pb_utils.Tensor("input_ids",
                                 encoded["input_ids"].astype(np.int32))
            attention_mask = pb_utils.Tensor("attention_mask",
                                 encoded["attention_mask"].astype(np.int32))

            responses.append(pb_utils.InferenceResponse(
                output_tensors=[input_ids, attention_mask]
            ))
        return responses
▸ Tokenizer pair format is model-specific
The exact separator and format for query/passage pairs is determined by how NV-RerankQA-Mistral-4B was trained. Check the model card at NGC for the exact prompt format. Some rerankers use "query: {q} document: {p}", others use native tokenizer pair encoding. The tokenizer_config.json extracted from NIM will contain the right settings — read it before writing your tokenizer backend.

▸ Output tensor name: verify before deploying
The output tensor name (sentence_embedding, pooler_output, logits, score) depends on how the specific model was compiled into the TRT engine. Always check the actual engine's config.json to confirm. Using the wrong name causes Triton to throw a tensor shape mismatch error at startup. The safe approach: run the extraction job, read the config.json, then write the config.pbtxt.
06 — Triton Ensemble Pipeline · live

Chain models into one logical call.

A Triton Ensemble chains multiple models into a single logical pipeline. It is configuration-only — no code, no versioned artifacts, just a config.pbtxt that wires outputs of one model to inputs of the next. From the client's perspective, you call the ensemble once with raw text; Triton internally executes tokenizer → engine → postprocess and returns the final result.

▸ Live: requests flowing through the embedding ensemble

Each dot is one inference request entering the ensemble. ▸ embed-pipeline (top): raw text → tokenizer → TRT engine → L2 normalize → 1024-d vector. ▸ rerank-pipeline (bottom): (query, passage) pair → tokenizer → TRT engine → sigmoid → relevance score. Dots glow brighter inside the two compute-heavy stages (tokenizer, TRT engine) to indicate where most of the work happens. Both ensembles share the same Triton process and the same L40S — but each request traverses only its own pipeline.

▸ Postprocess backend · L2 normalization

python · embed-postprocess/1/model.pyimport numpy as np
import triton_python_backend_utils as pb_utils


class TritonPythonModel:
    def initialize(self, args): pass

    def execute(self, requests):
        responses = []
        for request in requests:
            raw = pb_utils.get_input_tensor_by_name(
                      request, "sentence_embedding").as_numpy()

            # L2 normalize: embedding / ||embedding||₂
            # Required for cosine similarity to work as a dot product in pgvector
            norms = np.linalg.norm(raw, axis=1, keepdims=True)
            norms = np.maximum(norms, 1e-12)   # avoid div-by-zero
            normalized = (raw / norms).astype(np.float32)

            responses.append(pb_utils.InferenceResponse(
                output_tensors=[pb_utils.Tensor("embedding", normalized)]
            ))
        return responses

▸ Ensembles are free latency-wise
Triton chains the steps in-process, in shared memory — there is no network hop between tokenizer and engine. The animation above visualizes the logical stages; in reality, output tensors of one step are passed by-reference to the next. The whole pipeline runs as one inference call from the client's perspective.
07 — Unified Triton Deployment

One pod, one GPU claim, both pipelines.

With the model repository built on the PVC, the Triton deployment is straightforward: a single Deployment, one container, one GPU resource limit. Both embed-pipeline and rerank-pipeline load at startup; restarts skip extraction entirely and re-load from PVC in roughly 60 seconds.

▸ Complete deployment YAML

yaml · triton-deployment.yamlapiVersion: apps/v1
kind: Deployment
metadata:
  name: triton-small-models
  namespace: fhlb-ai-platform
  labels:
    app: triton-small-models
    fhlb.com/role: inference-embedding-reranker
spec:
  replicas: 1
  selector:
    matchLabels: {app: triton-small-models}
  template:
    metadata:
      labels: {app: triton-small-models}
    spec:
      serviceAccountName: nim-sa

      # Init container: symlink the extracted engines into the model repo
      initContainers:
      - name: setup-model-repo
        image: harbor.fhlbdm.internal/ai-platform/alpine-utils:latest
        command: ["/bin/sh", "-c"]
        args:
        - |
          EMBED_ENGINE=$(find /model-repo/nim-embedding/cache -name "rank0.engine" | head -1)
          RERANK_ENGINE=$(find /model-repo/nim-reranker/cache -name "rank0.engine" | head -1)
          echo "Embedding engine: $EMBED_ENGINE"
          echo "Reranker engine:  $RERANK_ENGINE"
          mkdir -p /model-repo/triton-models/embed-engine/1
          mkdir -p /model-repo/triton-models/rerank-engine/1
          ln -sf "$EMBED_ENGINE"  /model-repo/triton-models/embed-engine/1/rank0.engine
          ln -sf "$RERANK_ENGINE" /model-repo/triton-models/rerank-engine/1/rank0.engine
          for f in embed-tokenizer rerank-tokenizer embed-pipeline rerank-pipeline \
                   embed-engine rerank-engine embed-postprocess rerank-sigmoid; do
            [ -f "/model-repo/triton-models/$f/config.pbtxt" ] && \
              echo "OK: $f" || echo "MISSING: $f/config.pbtxt"
          done
        volumeMounts:
        - {name: model-repo, mountPath: /model-repo}

      containers:
      - name: triton
        image: harbor.fhlbdm.internal/ai-platform/triton-server:24.11-trtllm
        args:
        - tritonserver
        - --model-repository=/model-repo/triton-models
        - --model-control-mode=explicit
        - --load-model=embed-pipeline   # dependencies auto-load
        - --load-model=rerank-pipeline
        - --http-port=8000
        - --grpc-port=8001
        - --metrics-port=8002
        - --log-verbose=1
        ports:
        - {containerPort: 8000, name: http}
        - {containerPort: 8001, name: grpc}
        - {containerPort: 8002, name: metrics}
        resources:
          limits:
            nvidia.com/gpu: "1"     # ONE GPU — both engines share it
            memory: "24Gi"
            cpu: "8"
          requests:
            nvidia.com/gpu: "1"
            memory: "16Gi"
            cpu: "4"
        env:
        - {name: CUDA_VISIBLE_DEVICES, value: "0"}
        volumeMounts:
        - {name: model-repo, mountPath: /model-repo}
        readinessProbe:
          httpGet: {path: /v2/health/ready, port: 8000}
          initialDelaySeconds: 60
          periodSeconds: 10
          failureThreshold: 18   # 3 mins for both models to load
        livenessProbe:
          httpGet: {path: /v2/health/live, port: 8000}
          initialDelaySeconds: 30
          periodSeconds: 30

      nodeSelector:
        fhlb.com/gpu-role: "small-models"
      tolerations:
      - {key: nvidia.com/gpu, operator: Exists, effect: NoSchedule}

      volumes:
      - name: model-repo
        persistentVolumeClaim: {claimName: triton-model-repo-pvc}

▸ Restart behavior
All artifacts live on the PVC, so pod restarts skip extraction entirely. Triton re-mmaps each rank0.engine from the symlinked location and re-loads its Python backends from disk. Cold-start time: ~60 seconds for both pipelines. The readinessProbe failure threshold of 18 (3 minutes) absorbs occasional slow loads — node memory pressure, PVC contention, or first-touch page-in.

▸ The "is not ready" diagnostic chain
When Triton reports a model "is not ready", check in this order: ① engine file exists at the symlink target, ② engine was compiled on the same GPU type, ③ tensor names in config.pbtxt match config.json from the engine, ④ Python backend's EXECUTION_ENV_PATH points to a real conda env with transformers installed, ⑤ max_batch_size in config.pbtxt is feasible given current VRAM headroom.
09 — Spring AI Integration · keeping upstream code untouched

Adapt at the bean boundary; everything above stays the same.

Triton uses the Kserve v2 inference protocol, not the OpenAI embedding format. You need a small adapter layer in Spring Boot to translate between the Spring AI EmbeddingModel interface and the Triton v2 HTTP API. The key benefit: once this adapter is written, every component above it in the RAG pipeline is unchanged.

▸ Custom Triton embedding client (Spring AI bean)

java · TritonEmbeddingModel.java// Spring AI EmbeddingModel adapter for Triton v2 protocol
@Component
public class TritonEmbeddingModel implements EmbeddingModel {

    private final RestClient restClient;
    private final String tritonUrl;

    public TritonEmbeddingModel(@Value("${triton.base-url}") String tritonUrl) {
        this.tritonUrl = tritonUrl;
        this.restClient = RestClient.builder()
            .baseUrl(tritonUrl)
            .defaultHeader("Content-Type", "application/json")
            .build();
    }

    @Override
    public EmbeddingResponse call(EmbeddingRequest request) {
        List<Embedding> embeddings = request.getInstructions().stream()
            .map(text -> embedSingle(text.getText()))
            .map(vector -> new Embedding(vector, 0))
            .toList();
        return new EmbeddingResponse(embeddings);
    }

    private float[] embedSingle(String text) {
        // Build Triton v2 inference request
        var requestBody = Map.of(
            "inputs", List.of(Map.of(
                "name",     "text",
                "shape",    List.of(1, 1),
                "datatype", "BYTES",
                "data",     List.of(text)
            ))
        );

        var response = restClient.post()
            .uri("/v2/models/embed-pipeline/infer")
            .body(requestBody)
            .retrieve()
            .body(new ParameterizedTypeReference<Map<String, Object>>() {});

        // Extract float[] from Triton response
        var outputs = (List<Map<?, ?>>) response.get("outputs");
        var data = (List<Double>) outputs.get(0).get("data");
        float[] vector = new float[data.size()];
        for (int i = 0; i < data.size(); i++) vector[i] = data.get(i).floatValue();
        return vector;
    }
}

▸ Reranker in Spring AI RAG Advisor

java · TritonReranker.java// Wraps Triton rerank-pipeline for Spring AI RAG
@Component
public class TritonReranker {

    private final RestClient restClient;

    /**
     * Re-ranks retrieved documents by relevance to the query.
     * Calls rerank-pipeline on Triton for each (query, passage) pair.
     */
    public List<Document> rerank(String query, List<Document> candidates) {
        return candidates.stream()
            .map(doc -> {
                double score = rerankSingle(query, doc.getContent());
                return Map.entry(doc, score);
            })
            .sorted(Map.Entry.<Document, Double>comparingByValue().reversed())
            .map(Map.Entry::getKey)
            .limit(5)  // top-5 after reranking
            .toList();
    }

    private double rerankSingle(String query, String passage) {
        var requestBody = Map.of(
            "inputs", List.of(
                Map.of("name", "query",   "shape", List.of(1,1),
                       "datatype", "BYTES",  "data", List.of(query)),
                Map.of("name", "passage", "shape", List.of(1,1),
                       "datatype", "BYTES",  "data", List.of(passage))
            )
        );
        var response = restClient.post()
            .uri("/v2/models/rerank-pipeline/infer")
            .body(requestBody)
            .retrieve()
            .body(new ParameterizedTypeReference<Map<String, Object>>() {});

        var outputs = (List<Map<?,?>>) response.get("outputs");
        var data = (List<?>) outputs.get(0).get("data");
        return ((Number) data.get(0)).doubleValue();
    }
}
▸ No upstream code changes required
The TritonEmbeddingModel bean implements Spring AI's EmbeddingModel interface. The VectorStore, QuestionAnswerAdvisor, and RAGAdvisor all consume EmbeddingModel — they are completely unaware of whether the backing implementation is a NIM pod, a Triton ensemble, or an OpenAI-compatible endpoint. The only change from your current setup: swap the @Bean EmbeddingModel in your config class to return a TritonEmbeddingModel instead of the OpenAI client. Same shape, same contract, different backing service.
